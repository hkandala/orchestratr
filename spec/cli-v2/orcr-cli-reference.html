The orcr command reference

A ground-up simplification of the CLI: fewer verbs, one clear placement model for herdr sessions / workspaces / tabs / panes, and first-class visibility into every agent running in herdr — including ones orcr didn't start. 1 · The mental model

  1. orcr spawns real agent TUIs in your real herdr session. Every agent is a visible tab you can attach to in one keystroke. No hidden side session — the herdr UI is the viewer.
  2. orcr sees everything herdr sees. orcr ls lists agents you started by hand in a herdr pane right alongside orcr-managed ones, and send / wait / out / attach work on them too (with an honest, machine-readable downgrade of guarantees — see contract levels).
  3. Two independent axes: --cwd is where the agent process runs; --workspace is where its tab appears in herdr. Defaults: cwd = your cwd, workspace = derived from cwd. Neither ever silently changes the other.
  4. run waits and prints the answer. It's the better -p for every harness. Fan-out is explicit: --detach.
  5. Files are the inter-agent API. Prompts and responses are markdown files under runs/<id>/; after done the response file always exists and its provenance (file | transcript | scrape) is recorded.
  6. One durable job primitive: loop. Run an agent on a cron, an interval, or once at a time — until a condition. Goals, judges, and workflows are recipes built from run / send / wait / out — not verbs.
  7. Nine core commands (run ls show send wait out attach kill loop), three advanced (adopt daemon top), three plumbing (status is the health probe; gc, events).

2 · Placement: sessions, workspaces, tabs, panes, cwd

herdr's hierarchy is Session → Workspace → Tab → Pane → Agent. orcr's mapping:

herdr conceptwhat it isorcr's stance
Sessiona persistent herdr server namespace (own panes, sockets, state)orcr targets exactly one session per invocation: the session you're in when inside herdr, else config herdr.session — which defaults to herdr's default session, not a hidden side session. Override: --session / ORCR_SESSION. One config line (session = "orcr") restores full isolation.
Workspaceproject container, typically one per repo; has a default cwd; herdr's sidebar rolls up agent attention per workspacethe placement unit. Resolved from the agent's cwd by default (git-root first — order below). --workspace <id|label|new> overrides.
Taban addressable layout unit inside a workspace one tab per agent — labeled with the agent's name/id, prefixed by the parent's name for fan-outs (impl/a8). Predictable, attachable, closable.
Panea real terminalthe agent's TUI is the tab's single pane. --split right|down opts into splitting the caller's tab instead (side-by-side supervision). pane_id stays the only stable low-level handle.
Agenta process herdr recognizes in a pane (working / idle / blocked / done / unknown)completion detection rides these states (working → idle, then the response file is finalized). Foreign agents are first-class read/steer targets.
Worktreeherdr-managed git worktree bound to a workspace run --worktree [<branch>] provisions one and runs the agent inside it: cwd becomes the worktree path (conflicting --cwd is an error), placement goes to the worktree's workspace.
Workspace ≠ cwd. A herdr workspace has a default cwd but it is a UI container, not a directory. orcr keeps the axes separate: --cwd controls the process, --workspace controls the tab's home. When you set only --cwd, the workspace is derived from it; when you set only --workspace, the cwd is still your own.

Where a new agent lands — the context matrix

orcr detects its calling context from env: HERDR_* vars mark a herdr-managed pane; ORCR_* vars mark an orcr-spawned agent. Defaults per context:

you are…sessionworkspacetabcwd
a human (or agent) in a plain terminal config herdr.session (default: herdr's default). Server not running → orcr starts it headless and reports herdr_server: started; attach anytime with plain herdr. resolved from cwd (order below); created if none matches new tabcaller's cwd
a human at a herdr pane current sessioncurrent workspacenew tabpane cwd

a foreign agent in a herdr pane running orcr current sessioncurrent workspacenew tabpane cwd

Focus is never stolen in any context — --focus is always opt-in. Every spawn result reports the full resolved placement ({session, workspace, tab, pane}) plus workspace_resolution, so scripts never guess.

Workspace resolution order

  1. Explicit --workspace <id|label> (or new for a fresh one).
  2. Inside herdr with no --cwd override → the caller's current workspace.
  3. Existing workspace whose cwd equals the git worktree root of the agent's cwd (paths compared canonically — symlinks resolved).
  4. Existing workspace whose cwd equals the exact cwd.
  5. Nearest ancestor workspace, only if it lies within the same git worktree.
  6. Otherwise: create a workspace at the git root (or the cwd when not in git), labeled after the directory.

The matched rule ships in every result as workspace_resolution: explicit | current | git_root | exact_cwd | ancestor | created. This keeps monorepos from collapsing into one giant workspace and repeated spawns from spraying new ones.

Ownership: how sharing your session stays safe

  • Every workspace, tab, and pane orcr creates carries a durable machine-readable marker (pane env + herdr agent metadata with orcr as source).
  • gc, idle reaping, auto-close, and kill --tree only ever touch marked resources. Your hand-made panes are structurally out of reach.
  • Destructive operations on foreign targets always require --force.
  • Prefer total isolation anyway? [herdr] session = "orcr" in config — one line, everything else works identically.

3 · Targets, ids & contract levels

formexamplemeaning
a<N>a7orcr-managed agent — monotonic counter, never reused, stable in history forever
l<N>l2loop (the durable job)
nameimpla live --name label; typed-id grammar (^[al]\d+$, :t\d+, the word new) is reserved and rejected as a name — and a name held by a live agent can't be reused until it ends
herdr targetcodex-2a foreign agent, addressed by its herdr identity (terminal id / agent label), passed through as-is
a7:t2turn sugar, accepted by out and show (same as --turn 2)
canonical prefixorcr:a7 · name:impl · herdr:codex-2unambiguous forms for scripts — resolution never guesses across namespaces

Bare targets resolve orcr id → orcr name → herdr target; any collision is an error listing every match with its canonical form (exit 6 family, code ambiguous_target). Scripts should use canonical prefixes for anything destructive.

Contract levels

The same verbs work on every target, but the guarantees differ — and the difference is machine-readable, never silent. Every JSON row/object carries origin, contract, and a capabilities map:

contractwhoyou get
managedagents orcr spawnedeverything: run-dir files, turn tracking, lineage, response guarantee, steer/turn semantics, clean kill
adoptedforeign agents after orcr adopt full contract from the next turn onward; history before adoption stays untracked (orcr never fabricates turns it didn't observe)
foreignanything herdr detects that orcr didn't start read/steer/wait/attach best-effort: out falls back to transcript adapter or pane scrape (source recorded), wait rides herdr agent states, no run-dir, no turns, kill needs --force

Commands that need a missing capability fail with capability_unavailable + the exact missing key — for scripts that must not degrade, out --require-source file turns silent fallbacks into loud errors.

4 · Global conventions

  • Output: concise human text on a TTY; with --json, exactly one envelope object on stdout — {"ok":true,"result":…} or {"ok":false,"error":{"code","message","details"}} — logs on stderr. Exception: events --follow --json streams NDJSON.
  • Exit codes: 0 ok · 2 environment (herdr missing, config, version skew) · 3 timeout · 4 blocked · 5 killed · 6 not found · 7 state conflict · 1 other.
  • Stable error codes in every JSON error: ambiguous_target, state_conflict (with current_status + valid next commands), capability_unavailable, herdr_unreachable, placement_failed, limit_exceeded, lost_pane, lost_session.
  • Durations require units: 45s, 20m, 3h, 30d. Bare numbers are rejected with a hint — never silently interpreted. Milliseconds never appear (herdr's ms timeouts stay internal).
  • Placement echo: every command that creates or moves anything reports the resolved {session, workspace, tab, pane}.
  • Feature probe: status --json includes version, schema_version, and a features list, so skills and SDKs detect capabilities without try-and-fail.
  • Future-proofing: JSON identity objects reserve host (always local today) so remote hosts can arrive without breaking shapes.

5 · Command reference

core the nine commands the skill teaches · advanced real but rare · plumbing for SDKs and operators

orcr run core

Spawn an agent on any harness, wait for its answer, print it. The better -p for every harness.

Synopsis

orcr run -a <harness> [-p <text> | -f <file|->]
         [--name <label>] [--model <m>] [--effort <e>]
         [--cwd <dir>] [--worktree [<branch>]]
         [--session <s>] [--workspace <id|label|new>] [--split right|down] [--focus]
         [--keep] [--timeout <dur>] [--detach|-d]
         [--parent <id>] [--dry-run] [--mode tui|exec] [--json]

Behavior

  • Waits by default: blocks through the first turn, prints the response body, exits by outcome (0 done · 3 timeout · 4 blocked · 5 killed). Completion means the response file is finalized — herdr's idle signal alone is never enough.
  • --detach/-d returns immediately and prints the new id — the fan-out path. --json always carries the id, waiting or not.
  • Auto-closes the tab after the response is finalized (never before), recording close_reason. --keep leaves the agent idle for follow-up send --turns; kept agents reap after config idle_reap.
  • Placement per §2; the result reports it in full, plus herdr_server: existing|started|failed when auto-start was involved.
  • --worktree: provision a herdr worktree, run there (--cwd alongside it is an error).
  • --parent must name a live managed/adopted agent in the same session; depth/tree caps and cycles are rejected before spawning. Inside an orcr agent, lineage is automatic via env.
  • --dry-run: resolve placement, validate everything, spawn nothing — the JSON shows exactly what would happen.
  • --mode exec (advanced): headless harness invocation for API-billed or ultra-short tasks — still inside a herdr pane, still visible, registered to herdr with orcr as the reporting source.
  • Every spawn injects the env contract (ORCR_ID / ORCR_PARENT / ORCR_DEPTH / ORCR_STORE / ORCR_OUT) so nested orcr run calls self-assemble into a tree.

Examples

# ask codex, get the answer, done
orcr run -a codex -p "review the diff in this repo" --cwd ~/proj

# fan out three ways, gather later
orcr run -a claude -p "implement the parser"  --name impl -d --keep
orcr run -a codex  -p "write property tests"  --name tests -d
orcr run -a pi     -p "draft the docs"        --name docs -d
orcr wait impl tests docs --timeout 20m && orcr out impl --recursive --format path
JSON result shape

    "contract": "managed", "status": "done", "host": "local",

    "response": { "text": "…", "path": "…/runs/a7/001-response.md", "source": "file" },

} }

orcr ls core

One list for everything alive in the session — orcr-managed and foreign. Replaces v1's ps, tree, and history.

Synopsis

orcr ls [--tree] [--all] [--status <s>] [--managed|--foreign]
        [--harness <h>] [--since <dur>] [--json]

Behavior

  • Default: live agents in the target session, merged. The ORIGIN column (orcr / herdr) is always shown; foreign rows display their herdr target, never an a<N>-shaped id.
  • --tree renders lineage (foreign agents are roots); --all adds ended agents (the old history); --status blocked is the "who needs me" filter. Filters compose.
  • JSON rows carry origin, contract, and capabilities — scripts filter on those instead of assuming a uniform contract.

Examples

orcr ls                        # everything alive here, one table
orcr ls --tree                 # lineage view
orcr ls --status blocked       # agents waiting on a human
orcr ls --all --since 2h       # everything, including ended, last two hours
orcr ls --json | jq '.result.agents[] | select(.origin=="orcr") | .id'

orcr show core

Everything about one agent, as one object.

Synopsis

orcr show <target | target:tN> [--json]

Behavior

  • Identity, origin + contract + capabilities, status, placement, cwd, model/effort, timings, exit/close reasons, turns with prompt/response paths and sources, children ids, parent.
  • Foreign targets: everything herdr knows (state, pane, process) plus the capability map — no invented turns or history.
  • ls rows and show objects share one schema (show = the expanded form), so SDK types don't fork.

orcr send core

Message a live agent: steer it mid-turn, or hand a kept agent its next turn.

Synopsis

orcr send <target> [<text> | -f <file|->] [--steer | --turn] [--raw]
          [--wait] [--json]

Behavior

  • To a working agent = steer the current turn: recorded as NNN-prompt.K.md, still exactly one response, and completion re-arms — the turn ends only on the next stable working→idle after the last input. Blocked agents accept steers too (that's how you unstick them).
  • To an idle kept agent = the next turn (NNN+1-prompt.md).
  • Bare send resolves by live state and says what it did (steered a7 (working)). Scripts and the skill pin intent with --steer / --turn; a mismatch is exit 7 with current_status and the valid alternatives in details.
  • Foreign targets: works when herdr recognizes the pane as an agent. A pane that is not a recognized agent (say, a bare shell) requires --raw — orcr won't type into arbitrary terminals by default.
  • Done/ended target → exit 7; unknown → exit 6.

Examples

orcr send impl --steer "also handle escaped quotes"
orcr send impl --turn "now run the full test suite" --wait
orcr send codex-2 "looks good, continue"      # foreign, herdr-recognized

orcr wait core

Block until agents finish. The fan-in half of fan-out.

Synopsis

orcr wait <target...> [--any] [--tree] [--timeout <dur>] [--json]

Behavior

  • All-of by default; --any returns on the first completion; --tree waits on a node's live descendants too.
  • Managed targets complete when the response file is finalized. Foreign targets ride herdr states: an already-idle foreign target returns immediately (noted in the result); otherwise orcr waits for the next stable working→idle.
  • Timeout → exit 3 with completed / pending / blocked buckets; a blocked target → exit 4 with block_reason when herdr exposes one. If a pane or the session vanishes mid-wait, you get lost_pane / lost_session promptly — never a hang until timeout.

orcr out core

Read an agent's answer — the response contract, not a log stream.

Synopsis

orcr out <target | target:tN> [--turn N] [--recursive]
         [--format body|path|json] [--require-source file|transcript]

Behavior

  • Default: the latest response body. --turn N / :tN select a turn; --recursive walks descendants depth-first; --format path prints id name path lines for piping.
  • Provenance is explicit: each item reports source: file | transcript | scrape. Foreign targets fall back to the harness transcript adapter, else a pane scrape — --require-source makes scripts fail loudly instead of consuming low-confidence text.
  • Raw screen content is deliberately out of scope — that's herdr pane read.

orcr attach core

Go look at it. The moat in one verb.

Synopsis

orcr attach <target> [--takeover]

Behavior

  • Inside a herdr client: focuses the agent's tab. In a plain terminal: hands this terminal to the pane (herdr's attach; detach returns you here).
  • Default is observe; --takeover claims input — and is required for foreign targets, so you never accidentally type into someone's session.

orcr kill core

Stop execution: graceful per-harness exit, then close the pane.

Synopsis

orcr kill <target...> [--tree] [--force] [--json]

Behavior

  • Managed: run the harness's graceful shutdown recipe (~5s deadline), then close the orcr-owned pane and tab. Status killed, exit_reason recorded. The run dir and history remain.
  • --tree kills descendants bottom-up — managed descendants only by default; foreign nodes in the tree are skipped and listed. Crossing into them takes --force.
  • Foreign targets always require --force (orcr didn't start them), and the result lists exactly which panes were closed vs skipped.
  • Stopping a loop is kill l2; deleting its definition is loop rm l2.

orcr loop core

The one durable job primitive: run an agent on a cadence — or once at a time — until a condition. It survives your shell, because the daemon owns it.

Synopsis

orcr loop add ("<cron>" | --every <dur> | --once-at <time>)
              -a <harness> (-p <text> | -f <file>)
              [--max <n>] [--until <regex>] [--key <k>]
              [--catchup skip|once] [--expires <dur>] [run-flags…]
orcr loop ls | show <id> | pause <id> | resume <id> [--once-at <time>] | rm <id>

Behavior

  • Three cadence forms: five-field cron (stored UTC, confirmations echo local and UTC), --every intervals (measured from tick completion), and one-shot --once-at (RFC3339 or friendly forms). The once is in the flag name on purpose — a one-shot is a loop that fires exactly once, and the confirmation says so ("fires once at …, then ends"). Re-arm a fired one-shot with resume --once-at.
  • --max and --until <regex> (matched against each tick's response file) are termination policy. --every 20m --max 20 --until "ALL PASS" is the durable fix-until-green loop.
  • Each tick spawns a fresh agent with the loop's prompt and run-flags; ticks show up in ls --tree under the loop node l<N>. There is no built-in judge — a tick that needs one spawns it (the tick agent has the CLI too), or you use the worker/judge recipe in §7.
  • -f prompt files are re-read every tick — edit a running loop's instructions without touching it.
  • --key makes creation idempotent: retrying add --key nightly-triage returns the existing l<N> instead of a duplicate (agents retry; loops shouldn't multiply).
  • Missed ticks (laptop asleep, daemon down): --catchup skip (default) or once. Recurring loops run until removed; --expires opts into auto-expiry. Creation always prints the cadence in words and the exact cancel command.

Examples

orcr loop add "0 9 * * 1-5" -a claude -p "triage new issues" --key triage
orcr loop add --every 20m -a claude -f fix-tests.md --max 20 --until "ALL PASS"
orcr loop add --once-at "tomorrow 07:00" -a codex -p "run the release checklist"

orcr adopt advanced

Promote a hand-started herdr agent to orcr-managed.

Synopsis

orcr adopt <herdr-target> [--name <label>] [--json]

Behavior

  • Assigns an a<N> id immediately; the full contract (turn files, response guarantee, lineage) starts with the next orcr-initiated turn after the agent is idle. Until then the contract reads adopted with pending: true.
  • Nothing is back-filled: pre-adoption output stays untracked. Adopting mid-turn never splices into a turn orcr didn't start.
  • Foreign targets already work degraded without adoption — adopt when you want stable ids, run-dir files, and multi-turn tracking on an agent you started by hand.

orcr status plumbing

One health probe for the whole stack.

Synopsis

orcr status [--json]

Behavior

  • Reports: orcr version + schema_version + features list, store path and lock state, daemon (running / pid / pending loops + next fire), herdr binary + version, target session and server reachability, live agent counts by status (blocked agents named), reconciliation drift.
  • This is the daemon status — there is no separate daemon status to remember.

orcr daemon advanced

Control the background supervisor (it normally manages itself).

Synopsis

orcr daemon start | stop

Behavior

  • Auto-starts on demand — the first loop add, detached-run timeout, or kept-agent reap that needs it. Manual control is for operators.
  • The daemon owns: loop execution, idle reaping, timeout enforcement for detached runs, reconciliation after herdr restarts — and is the single writer for durable job state (foreground verbs coordinate through it when it's up, a store lock when not).
  • Named daemon deliberately: herdr server is the substrate, orcr daemon is the supervisor. Two words, two processes.

orcr top advanced

A live tree TUI for orcr state — turns, lineage, loops, blocked waits.

Synopsis

orcr top [--pane]

Behavior

  • For SSH boxes and plain terminals where the herdr UI isn't in front of you; inside herdr, --pane opens it as a split. Keys: Enter attach · s send · k/K kill · o open response · q quit.
  • Nothing auto-opens it anymore — agents now live in your visible session, so herdr itself is the default viewer (v1's auto-viewer pane is gone).

orcr gc plumbing

Reconcile recorded state with herdr reality.

Synopsis

orcr gc [--dry-run] [--json]

Behavior

  • Diffs the store against the live session: orcr-marked panes unknown to the store are closed; recorded agents whose panes vanished are marked lost; dead session registrations are dropped. Reports every action (start with --dry-run).
  • Structurally incapable of touching unmarked (user-created) panes, tabs, or workspaces.

orcr events plumbing

The append-only event feed — what the TUI and SDKs subscribe to.

Synopsis

orcr events [--follow] [--json]

Behavior

  • Spawns, status changes, turns, steers, loop ticks, reconciliation actions. --follow --json streams NDJSON ({type, id, time, payload} per line) — the one exception to the single-object rule.

6 · What was cut — and its replacement

The v1 surface had 20 verbs; v2 has 15 (9 core). The test for every cut: can it be rebuilt from the outside with the remaining primitives? If yes, it's a recipe or SDK helper, not a command.

loop + schedule (two verbs)one loop primitive: --every/cron cadence, --once-at one-shots, --max/--until terminationboth were "run an agent on a cadence with a stop rule"; one noun, honestly named for the headline case, with one-shots explicit in the flag

The goal recipe (worker + independent judge)

orcr run -a claude -p "$(cat goal.md)" --name worker --keep -d
for i in 1 2 3 4 5; do
  orcr wait worker --timeout 15m
  verdict=$(orcr run -a codex -p "Judge against goal.md. First line: PASS or FAIL: reasons.
$(orcr out worker)")
  case "$verdict" in PASS*) break;; esac
  orcr send worker --turn "Judge feedback — address and retry: ${verdict#FAIL:}"
done
orcr kill worker

Ships as a documented recipe and an SDK helper — outside the CLI, inside the product.

Deliberately not added

  • No coordination layer (orca-style inboxes, asks, decision gates, task boards). orcr is the terminal/session substrate; coordination composes on top via files and the SDK, against stable ids, lineage, run dirs, and the event stream.
  • No remote hosts, no permission broker — yet. The JSON shapes reserve room (host, capability maps, blocked semantics) so both can land without new verb sprawl.

7 · SDK: the six workflow patterns

The SDK (TypeScript + Python) is a thin typed wrapper over orcr … --json — no private protocol, nothing you couldn't do in shell. The surface used below:

SDKwrapsnotes
orcr.run(opts) → Agentrunwaits and resolves with the response by default; detach: true resolves immediately with a live handle; keep: true for multi-turn workers
agent.out() / .turn(p) / .steer(p) / .wait() / .kill() / .show() out send wait kill show.turn() resolves with that turn's response
orcr.wait(agents, opts)wait {any, tree, timeout}
orcr.loop.add(opts) → Looploop add durable — the daemon owns it, your process can exit
orcr.withGroup(label, fn)env contractparents every spawn inside fn to one group node; kills orphans on throw

Every pattern below runs agents as real, visible herdr tabs — you can watch any step of any workflow live, attach, and steer it mid-flight. That composability question — "can a workflow do its steps, consolidate, then set up a loop and leave?" — is pattern 6b: everything before loop.add is your process; everything after belongs to the daemon.

1 · Classify-and-act

import { orcr } from "@orchestratr/sdk";

const routes = {
  bug:     { harness: "claude", prompt: (t) => `Reproduce, then fix this bug:\n${t}` },
  feature: { harness: "codex",  prompt: (t) => `Implement with tests:\n${t}` },
  docs:    { harness: "pi",     prompt: (t) => `Write the docs for:\n${t}` },
};

const cls = await orcr.run({
  harness: "claude", model: "haiku",           // cheap classifier
  prompt: `Classify as exactly one word — bug, feature, or docs:\n${task}`,
});
const route = routes[cls.response.text.trim().toLowerCase()] ?? routes.bug;
const result = await orcr.run({ harness: route.harness, prompt: route.prompt(task), cwd });
console.log(result.response.text);

2 · Fanout-and-synthesize

const angles = ["security", "performance", "API design", "test coverage"];

const reviewers = await Promise.all(angles.map(a => orcr.run({
  harness: "codex", cwd, detach: true, name: `rev-${a.replace(/\W+/g, "-")}`,
  prompt: `Review this repo strictly for ${a}. End with a ranked findings list.`,
})));

await orcr.wait(reviewers, { timeout: "20m" });
const reports = await Promise.all(reviewers.map(async r => `## ${r.name}\n${await r.out()}`));

const synthesis = await orcr.run({ harness: "claude", prompt:
  `Merge these review reports into one deduplicated, severity-ranked report:\n\n${reports.join("\n\n")}` });
console.log(synthesis.response.text);

3 · Adversarial verification

const worker = await orcr.run({ harness: "claude", cwd, keep: true, name: "worker",
                                prompt: taskPrompt });
let answer = worker.response.text;

for (let round = 0; round < 3; round++) {
  const verdicts = await Promise.all([1, 2, 3].map(() => orcr.run({
    harness: "codex", cwd,                      // different harness = independent eyes
    prompt: `Try hard to REFUTE this solution. First line: "REFUTED: <why>" or "HOLDS".\n\n${answer}`,
  })));
  const refuted = verdicts.filter(v => v.response.text.startsWith("REFUTED"));
  if (refuted.length <= 1) break;               // survives a majority of skeptics

  answer = (await worker.turn(
    `Independent verifiers refuted your solution:\n` +
    refuted.map(v => v.response.text).join("\n") + `\nAddress every point and answer again.`,
  )).text;
}
await worker.kill();

4 · Generate-and-filter

const generators = await Promise.all(Array.from({ length: 5 }, (_, i) => orcr.run({
  harness: i % 2 ? "codex" : "claude",          // mixed harnesses = diverse ideas
  detach: true, name: `gen-${i}`,
  prompt: `Propose 3 genuinely distinct approaches for: ${brief}. One per "## Idea" section.`,
})));
await orcr.wait(generators);

const ideas = (await Promise.all(generators.map(g => g.out()))).join("\n\n---\n\n");
const best = await orcr.run({ harness: "claude", prompt:
  `Candidate approaches:\n${ideas}\n\n` +
  `Dedupe near-identical ideas, score each against this rubric: ${rubric}. ` +
  `Return the top 2 verbatim with scores; list the discards in one line each.` });

5 · Tournament

const seeds = ["MVP-first", "risk-first", "user-first", "perf-first"];
const attempts = await Promise.all(seeds.map(angle => orcr.run({
  harness: "claude", detach: true, name: angle,
  prompt: `Design the solution ${angle}: ${brief}`,
})));
await orcr.wait(attempts);

let pool = await Promise.all(attempts.map(async a => ({ name: a.name, text: await a.out() })));
while (pool.length > 1) {                       // pairwise judge bracket
  const next = [];
  for (let i = 0; i < pool.length; i += 2) {
    const j = await orcr.run({ harness: "codex", prompt:
      `Which design better serves "${brief}"? First line exactly A or B, then why.\n\n` +
      `A (${pool[i].name}):\n${pool[i].text}\n\nB (${pool[i+1].name}):\n${pool[i+1].text}` });
    next.push(j.response.text.trimStart().startsWith("B") ? pool[i+1] : pool[i]);
  }
  pool = next;
}
console.log(`winner: ${pool[0].name}\n\n${pool[0].text}`);

6 · Loop until done — and 6b: set it up and leave

// 6 — foreground: keep rounds going until two consecutive rounds find nothing new
let dry = 0;
while (dry < 2) {
  const r = await orcr.run({ harness: "codex", cwd, prompt:
    `Hunt for bugs NOT already listed in findings.md; append new ones there. ` +
    `Reply exactly NONE if you found nothing new.` });
  dry = r.response.text.includes("NONE") ? dry + 1 : 0;
}

// 6b — the durable handoff: do the steps, consolidate, then LEAVE.
// Everything above ran in this process; the loop below outlives it.
await orcr.loop.add({
  every: "30m", harness: "codex", cwd,
  promptFile: "findings.md",                    // re-read every tick — steer by editing it
  until: "ALL RESOLVED", max: 48, key: "burn-down",
});
process.exit(0);   // daemon owns it now — check in later: orcr loop ls · orcr ls --tree

Patterns compose: a tournament can seed pattern 3's worker; any tick of a durable loop can itself run pattern 2, because the tick agent has the CLI too. The v1 goal verb is pattern 3 with one verifier.

9 · Design-review adjudication

The draft went through four independent design reviews, each with a different lens: LLM-caller ergonomics, human CLI conventions, systems correctness / herdr integration, and minimalism / extensibility. What they changed:

Adopted from review

  • Contract levels + capability maps on every target (three reviews rated silent managed/foreign polymorphism the top risk). Degradation is now always machine-readable, and --require-source exists.
  • Git-root-first workspace matching with canonical paths and a reported resolution reason (all four flagged exact/ancestor matching as monorepo-hostile).
  • One status — the separate daemon status was unanimously called a distinction users shouldn't learn.
  • Ownership markers for everything orcr creates; gc/reap/kill are structurally scoped to them. kill --tree stops at foreign nodes by default.
  • Foreign-input safety: send --raw gate for unrecognized panes; attach --takeover required for foreign targets.
  • Adoption semantics hardened: id now, contract from next idle + orcr-initiated turn, zero history reconstruction.
  • Lifecycle honesty: auto-close only after the response is finalized; lost_pane/lost_session fail fast instead of hanging; herdr_server: existing|started|failed reported on auto-start.
  • loop --key idempotency (retrying agents were creating duplicate jobs); --parent validity rules; required duration units; run --dry-run; feature probe in status --json; reserved host field.

Adjudicated against (with reasons)

  • "Default plain-terminal spawns to an isolated orcr session" (systems review). Rejected: visibility in the user's real session is the product's core promise; the blast-radius concerns are answered with ownership markers, force-gated foreign ops, and the one-line isolation config. The dissent is recorded — if markers prove insufficient in practice, this is the first decision to revisit.
  • "Make ls --json managed-only by default" (LLM-ergonomics review). Rejected: seeing everything is the moat; safety comes from origin/contract fields + capability errors + force gates, not from hiding rows.
  • "Rename out → logs" (two reviews). Rejected: out returns the final-answer contract, not a stream; logs would promise docker-style tailing that herdr pane read already owns. Discoverability is handled in --help ("read an agent's answer").
  • "Add an automation mode env-var that flips defaults". Rejected: two default-sets is how CLIs become unpredictable; instead, the skill teaches explicit flags (--json, pinned intent, canonical targets).

Post-review naming call

The durable primitive shipped in the reviewed draft as schedule; it was renamed loop afterward (owner's call): the headline use — iterate until a condition — is loop engineering, and the name should say so. The one place the name could mislead, one-shots, is defused by putting the semantics in the flag (--once-at) and in the confirmation output.

Full review transcripts live alongside the spec; every remaining open question is intentionally none.
