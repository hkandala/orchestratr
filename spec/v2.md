# orchestratr v2 — design specification

orchestratr (CLI: `orcr`) is a cross-harness orchestrator for AI coding agents, built on
[herdr](https://herdr.dev). This document is the complete v2 specification: problem, core
concepts, CLI, SDK, execution model, storage, and configuration. It supersedes the v1
surface described in `spec/01–10` (the execution machinery there — herdr driver rules,
profiles, store discipline — carries forward except where this document says otherwise).

Status: design locked pending final owner review; not yet implemented. Revision 3:
incorporates the findings of three independent spec reviews (systems correctness, API/DX,
completeness).

---

## 1 · Problem statement

Coding agents (Claude Code, Codex CLI, Pi, OpenCode) are single-player tools: one
terminal, one session, one human watching. Real work wants many of them — a reviewer
fanned out per concern, a worker iterating under a verifier, a nightly job that triages
issues — often spanning *different* harnesses, since each has different strengths and
each subscription has its own quota.

Running agents as headless API calls solves scale but loses three things that matter:

1. **Plan pricing.** Interactive TUI sessions bill against flat subscription plans;
   API-mode invocations bill per token. At orchestration scale the difference is the
   whole budget.
2. **Steerability & visibility.** A headless agent can't be watched mid-flight, nudged
   with a correction, or taken over when it goes sideways. A real TUI can.
3. **Uniformity.** Every harness has a different headless mode, different flags,
   different transcripts. A script that fans out across harnesses needs one interface.

orcr's answer: run every agent as a **real interactive TUI in a herdr-managed terminal
pane**, and expose the smallest possible command surface to spawn, message, await, read,
and stop them — from a shell, from any programming language, or from *inside another
agent*. herdr supplies the terminal substrate (sessions, panes, agent state detection,
attach-from-anywhere); orcr supplies identity, grouping, lifecycle, scheduling, and a
uniform cross-harness contract.

### Goals

- Agents as real TUIs: plan-pricing-safe, attachable, steerable, visible.
- One interface across harnesses (claude / codex / pi / opencode; extensible per-harness
  profiles).
- Extreme-minimal primitives that compose from any language — the CLI is the API; the
  SDK is a convenience, not a requirement.
- Agents can orchestrate agents: any orcr-spawned agent can call `orcr` itself; lineage
  and grouping assemble automatically.
- Safe at scale: global and per-harness concurrency caps (RAM protection), automatic
  lifecycle GC, one owned herdr session so the user's own workspace is never polluted.
- Organized at scale: hierarchical groups make a 200-agent workflow legible, operable
  (`wait`/`kill` by group), and visualizable in herdr's native UI.
- Durable scheduling: run any command on a cadence, surviving the caller's shell.

### Non-goals (v2)

Remote hosts · permission policies / read-only mode (v2 runs harnesses in their
bypass-permissions mode; every spawn result carries `"permissions": "bypass"`) ·
Windows · Python SDK · agent-to-agent coordination primitives (inboxes, decision gates,
task boards) · git worktree provisioning · a monitoring TUI (`herdr --session orcr` *is*
the dashboard).

## 2 · Architecture

```
you / a script / another agent
        │  (shell, or TS SDK over --json)
        ▼
  orcr CLI  ──► store (sqlite: agents · loops · loop_runs · events)
        │              ▲
        │              │ single writer for durable state (daemon lease / store lock)
        ▼              │
  orcr daemon ─────────┘   loops · GC mover/reaper · queue promotion · reconcile
        │
        ▼
  herdr (external binary, discovered — never embedded)
        └─ session "orcr" ─ workspaces (= level-1 groups) ─ tabs (= agents) ─ panes
                                └─ real TUIs: claude / codex / pi / opencode
        └─ transcript adapters read each harness's native session files
```

- **CLI** — all verbs; the `agent` verbs work with no daemon (see the daemon-down matrix,
  §6.6).
- **Daemon** — auto-started on demand; owns everything time-based. Holds a **lease**
  recorded in the store; foreground verbs mutate durable state only through atomic store
  transactions, and perform daemon duties inline only when the lease is absent/expired.
- **herdr driver** — shells the `herdr` CLI, parses its JSON envelopes. Discovery:
  config `herdr.bin` → `$ORCR_HERDR_BIN` → `$PATH`; missing → friendly pointer, exit 2.
- **Profiles** — one module per harness: launch argv (bypass flags, model/effort
  mapping), startup recipe, completion detection parameters, graceful-shutdown recipe,
  transcript adapter. Adding a harness = one file.
- **Store** — sqlite (WAL, `user_version = 2`); no run directories in v2. Opening a v1
  store (`user_version = 1`) is refused with a clear message and a rename instruction —
  no auto-migration.

## 3 · Core concepts

### 3.1 The owned session

All orcr agents live in one dedicated herdr session (default name `orcr`, config
`[herdr] session`). The user's daily herdr session never sees a subagent pane.

- First use auto-starts the session's server headless; `herdr --session orcr` opens the
  native UI over it — this *is* the monitoring dashboard.
- `orcr agent attach` wraps `herdr agent attach`, which streams a pane's terminal into
  the current terminal from anywhere — no session switching.
- **Store↔session binding.** Each store has a generated `store_id`. Every pane orcr
  creates carries `ORCR_STORE_ID` + `ORCR_ID` in its env — a durable ownership marker.
  On startup, a store *claims* its session (recorded in the store and probed via a
  marker pane env); if the session is already claimed by a different live `store_id`,
  orcr fails fast (exit 2) rather than fight over panes. Reconciliation (§6.5) closes
  only panes marked with **this** store's id; unmarked panes in the owned session (a
  shell you opened while debugging, another store's agents) are reported in
  `daemon status` as `foreign_in_owned_session` and never touched automatically.
- Foreign sessions: see §4.4 for the exact contract per verb.

Within the session, herdr's hierarchy is used as follows. (herdr facts: workspaces are
per-session, purely visual containers with a label and no cwd; only panes have a cwd
and a process.)

| herdr level | orcr's use |
| --- | --- |
| workspace | = the agent's **level-1 group segment**, displayed via the transform (§3.2): all of `phase_1.*` → workspace "Phase 1"; each loop's runs → one workspace per loop. Ungrouped agents → `default`. GC-parked agents → `idle`. herdr removes empty workspaces automatically. |
| tab | one per agent; label = remaining group path + name/id ("Review / File 1 · a42") |
| pane | the agent's TUI; cwd = caller's cwd or `--cwd`; `pane_id` is the stable internal handle |

### 3.2 Groups

Groups organize agents; they are **strings, not entities** — never created, never
deleted, no registry. A group exists exactly as long as some agent (live or historical)
carries it.

- **Grammar**: dot-separated segments, each `[a-z0-9_]+`. Limits: ≤ 6 segments, ≤ 64
  chars/segment, ≤ 256 chars total. Violations are rejected at spawn with a clear
  error, never truncated.
- **Display transform** (derives human labels from machine slugs; no stored metadata):
  underscores → spaces, words title-cased, dots → " / ".
  `phase_1.review.file_1` → **Phase 1 / Review / File 1**.
- **Inheritance — relative by default**: *effective group = `$ORCR_GROUP` prefix `.`
  explicit `--group`* (env alone if no flag; flag alone if no env; ungrouped if
  neither). A **leading `/` makes the argument absolute** — `--group /refactor` ignores
  any inherited prefix (`/` is outside the segment charset, so this is unambiguous).
  Every spawn exports the effective group as `ORCR_GROUP` to the agent, so its own
  spawns nest. Consequences:
  - a workflow script exports `ORCR_GROUP=refactor` once (or SDK `withGroup`) — every
    agent spawned inside, however deep, lands under `refactor.…`;
  - an agent in `refactor.phase_1` spawning with `--group file_1` yields
    `refactor.phase_1.file_1`; with no flag it inherits `refactor.phase_1` verbatim;
  - a loop sets the prefix to `<loop_name>.<run_ts>` per run (§4.2).
- **Scope is always echoed**: every spawn result and `ls` row carries the machine
  `group` (effective path) alongside `group_display`; TTY `ls` always shows the machine
  path so `wait`/`kill --group` arguments can be copied, not guessed.
- **Prefix operations**, matched on segment boundaries (`phase_1` never matches
  `phase_10`): `agent ls|wait|kill --group <prefix>`. Prefix arguments are absolute
  (never composed with `$ORCR_GROUP`) — filters and destructive scopes must be
  explicit.
- **Same name = same group, by definition.** No auto-suffixing. Scripts wanting per-run
  isolation stamp their own suffix (`--group "refactor.$(date +%s)"`).

### 3.3 Lifecycle (GC)

One policy replaces v1's auto-close/keep/reaper trio: `--gc auto|immediate|never`.
`--gc` governs **pane lifetime only** — history in the store is unaffected.

| mode | behavior |
| --- | --- |
| `auto` (default) | turn-complete and idle for `idle_after` (5m) → pane moved to the `idle` workspace (`parked`) → `kill_after` (10m) more → graceful kill, memory released. An inbound `send` at any point moves the agent back to its workspace and resets both clocks. |
| `immediate` | pane closed as soon as the first turn completes **and its final response has been captured** (two-phase: stable idle → transcript quiescence → response recorded → kill). The agent ends `done` (`exit_reason: completed`), not `killed` — `run --wait` on it exits 0. |
| `never` | exempt from parking and reaping; lives until an explicit `agent kill`. For pinned long-livers (heartbeat agents). |

A stuck-*working* agent never goes idle, so `--timeout <dur>` (config default `30m`)
remains the safety valve; on expiry the agent is killed (`exit_reason: timeout`).

**Interlocks** (all state transitions are versioned compare-and-swap on the agent row):
`send` cancels a pending park/reap atomically *before* delivering input; GC never moves
or reaps a pane with an active attach; completion capture and GC-kill are ordered (the
response is recorded before the pane dies); a parked agent's un-park happens inline in
the sending CLI, not the daemon.

### 3.4 Concurrency

- **Global cap** `concurrency.max` (default 25) — RAM protection; heavy TUIs at 100×
  will take a machine down.
- **Per-harness caps** beneath it (e.g. `claude = 10`, `pi = 50`); a spawn needs a free
  slot in both.
- Over-cap spawns get status `queued` (the id is allocated and printed immediately) and
  are promoted strictly FIFO by `queue_seq`. Promotion is an **atomic store
  transaction**: `queued → starting` succeeds only if the row is still queued *and* a
  capacity recount under the write lock shows free slots — no double-promotion, no
  cap overshoot, whether the promoter is the daemon or (lease absent/expired) the next
  CLI invocation. A `starting` row that makes no progress within a bound is reconciled
  to `failed`.
- `wait` on a queued agent waits through promotion and its first turn; `kill` on a
  queued agent dequeues it (`exit_reason: canceled`).
- Loops have a separate, unrelated knob: `--concurrency` caps concurrent *runs of that
  loop* (§4.2).

### 3.5 Identity, names & env contract

- Ids: agents `a<N>`, loops `l<N>` — monotonic counters, never reused. Loop runs:
  `l<N>.r<seq>` (monotonic per loop); each run also records its fire timestamp, which
  is what appears in the run's group path (§4.2).
- `--name`: a label unique among **live + queued** agents store-wide (dead names are
  reusable); needed for send-by-name (heartbeats). Typed-id shapes (`^[al]\d+$`,
  `r\d+`) are rejected as names.
- **Target resolution** (same algorithm for send/logs/wait/attach/kill): (1) exact
  typed id → the store; (2) otherwise a live-or-queued managed name. Ambiguity is
  impossible by construction (names are unique, id shapes are reserved); a dead
  historical name resolves to nothing — use the id (exit 6 with a hint listing recent
  dead matches). With `--session <s>`, resolution is herdr-target lookup in that
  session only — never mixed with owned-store lookup.
- Env injected into every agent pane and loop-run command:

  ```
  ORCR_ID        agent id (a7)                     [agents]
  ORCR_STORE_ID  owning store's identity            [agents]
  ORCR_GROUP     effective group (prefix for children)
  ORCR_PARENT    parent ref: a3 | l2.r17 (unset at root)
  ORCR_SESSION   the owned herdr session name
  ORCR_LOOP      loop id (l2)                       [loop-run commands + descendants]
  ORCR_RUN       run ref (l2.r17)                   [same]
  ```

  Agents spawned by a loop-run script get `parent = l2.r17` (`parent_kind: loop_run`)
  unless an agent is the direct spawner.

### 3.6 Completion discipline (internal, hard-won)

herdr reports agent state (`working / idle / blocked / unknown`). Completion is defined
per **input epoch**: every delivered input (the first prompt, every `send`) increments
the agent's `input_seq` *before* delivery. A turn is complete when, for the latest
epoch: `working` has been observed **after that epoch's delivery began** (small
per-harness grace window for fast turns), followed by **stable idle** — stability
defined per profile as a minimum idle duration *and* transcript quiescence. A first
idle without epoch-scoped working is never completion; an old idle can never satisfy a
newer send. `blocked` is epoch-scoped too (categories best-effort:
`question|limit|login|unknown`): it surfaces as status + exit 4 for waiting callers and
is clearable by `send` (state moves to working-pending; if the same blocked marker
persists past a delivery grace, it returns to `blocked`).

Other carried-forward herdr driver rules: input delivery is two calls (send-text →
~1s → enter — never one); `pane_id` is the only stable handle after launch; herdr
`--timeout` values are milliseconds and never leak into orcr's user surface.

## 4 · CLI

Three nouns. All verbs support `--json` (exactly one envelope object on stdout —
`{"ok":true,"result":…}` / `{"ok":false,"error":{code,message,details}}` — logs to
stderr). Exit codes: `0` ok · `2` environment · `3` timeout · `4` blocked · `5` killed
· `6` not found · `7` state conflict · `1` other. Durations always carry units
(`45s`, `20m`, `3h`); bare numbers are rejected.

### 4.1 agent

```
orcr agent run    -a <harness> [-p <text> | -f <file|->]
                  [--name <label>] [--group <path>] [--gc auto|immediate|never]
                  [--model <m>] [--effort <e>] [--cwd <dir>] [--timeout <dur>]
                  [--wait] [--session <s>] [--json]
orcr agent send   <id|name> [<text> | -f <file|->] [--expect idle|working|parked] [--json]
orcr agent logs   <id|name> [--last-response] [--tail <n>] [--follow] [--json]
orcr agent wait   <id|name>... | --group <prefix>  [--any] [--timeout <dur>] [--json]
orcr agent attach <id|name> [--takeover]
orcr agent kill   <id|name>... | --group <prefix>  [--force] [--json]
orcr agent ls     [--group <prefix>] [--all] [--status <s>] [--harness <h>]
                  [--session <s>] [--json]
```

**run** — **async: prints the new id (stdout carries only the id in non-JSON mode) and
returns**; a TTY also gets a stderr hint (`wait: orcr agent wait a7 · response: orcr
agent logs a7 --last-response · attach: orcr agent attach a7`). `--wait` blocks through
the first completion — including queue time — and prints the final response from the
transcript; if the turn completed but no final response can be identified, it exits 1
with `error.code: transcript_unavailable` (details include the agent id and status so
the caller can recover). Placement per §3.1, admission per §3.4, group per §3.2, gc per
§3.3. Auto-starts the owned session's server if needed
(`herdr_server: existing|started` in the result).

**send** — types the text into the agent's TUI and submits, whatever state the agent is
in (harness TUIs queue mid-turn input natively; there are no steer/turn modes). The
result always reports what happened: `delivered_while: working|idle|parked` +
`input_seq`. Automation that must not steer mid-turn passes `--expect idle` — a state
mismatch is exit 7 with `{current_status, expected}` and nothing is sent. Sending to a
parked agent un-parks it (atomically, before delivery). Ended target → exit 7 with the
terminal status in details; unknown target → exit 6.

**logs** — reads the harness's **native transcript** via the per-harness adapter
(structured turns, tool calls, token counts where available). `--last-response` prints
only the final assistant message and **fails loudly rather than guessing**: exit 1
`transcript_unavailable` when no final response is identifiable; exit 2
`transcript_unsupported` when the harness has no adapter (see the capability matrix,
§6.4). `--follow` streams. Works on hand-started agents when herdr detection reports a
compatible `agent_session`; ambiguous transcript candidates are an error listing the
candidates, never a silent pick.

**wait** — targets are ids/names, or `--group <prefix>` (membership = live + queued
managed agents under the prefix, **snapshotted at invocation**; ended agents excluded).
Per-target outcome: `completed` (latest epoch turn-complete — an already-complete idle
agent completes immediately) · `blocked` · `killed` · `lost` · `failed`. All-of
(default): returns when every target has an outcome or `--timeout` expires; exit 0 iff
all completed, else the highest-priority error among outcomes (blocked→4, killed→5,
lost/failed→1) or 3 on wait-timeout. `--any`: returns 0 on the first `completed`;
non-completable targets don't end the wait while at least one target can still
complete. JSON always returns every target as `{id, status, outcome, exit_reason}`
plus summary buckets.

**attach** — wraps `herdr agent attach`: streams the pane into the current terminal
from anywhere (inside or outside herdr), detach returns. Observe by default,
`--takeover` claims input.

**kill** — graceful per-harness shutdown recipe (~5s deadline) → pane close; status
ends `killed`; history remains. `--group` = subtree cleanup in one command; queued
agents are dequeued (`canceled`). Partial success exits 0 with `killed[]` and
`skipped[]` (already-ended) arrays. Foreign-session targets require `--force`.

**ls** — live + queued agents rendered as the group tree; headings show the display
label *and* the machine path. TTY columns: `ID NAME STATUS HARNESS GROUP AGE`.
`--all` includes ended agents (history — including every past loop run, queryable by
group). `--status blocked` = who needs a human. JSON rows are flat:
`{id, name, group, group_display, state, turn, harness, session, cwd, pane_id,
queue_position?, created_at, ended_at?, exit_reason?}`.

### 4.2 loop

```
orcr loop create ("<cron>" | --every <dur> | --once-at <time>)
                 [--name <n>] [--concurrency <n>] [--overlap queue|skip]
                 [--timeout <dur>] [--json] -- <command…>
orcr loop ls | logs <id> [--run <ref>] | pause <id> | resume <id> [--once-at <time>] | rm <id> [--kill-active]
```

Durable cron for **any command** — the `--` boundary captures an **argv array**,
executed directly (no shell). Want shell features? Say so:
`-- sh -c 'a && b'`. Creation echoes the parsed argv verbatim, the cadence in words
(local + UTC), and the exact cancel command. The command spawns agents via CLI/SDK like
any other caller; the loop owns *time only*: no harness flags, no prompts, no judge
logic, no stop-condition DSL.

- **Cadence**: five-field cron — stored **with the creating timezone** and evaluated in
  it (DST-correct: "9am weekdays" stays 9am), each occurrence persisted as a UTC
  `next_fire_at` · `--every <dur>` (measured from last run completion at
  `--concurrency 1`, from last fire otherwise) · `--once-at <time>` (fires once then
  ends; re-arm via `resume --once-at`). Fires missed while the machine slept or the
  daemon was down are skipped and logged.
- **Runs & grouping**: each run is `l<N>.r<seq>` (monotonic), started in its **own
  process group** (pid/pgid recorded). It executes with
  `ORCR_GROUP=<loop_name>.<run_ts>` — `run_ts` is the fire's unix seconds, bumped to
  the next unused second on collision, so group paths stay unique per run. The
  script's own `--group phase_1.review` nests to
  `nightly.1752300000.phase_1.review`. Level-1 = loop name → each loop gets its own
  workspace; `agent ls --all --group nightly` is the loop's full agent history.
- **Concurrency & overlap**: `--concurrency` caps concurrent runs (default 1). At the
  cap, `--overlap queue` (default) records **at most one pending fire** (`pending_due_at`
  — later fires coalesce into it; survives daemon restarts); `skip` drops the fire with
  a log line. When a run exits and a pending fire is recorded, it fires immediately.
- **Per-run timeout**: TERM to the run's process group → grace → KILL → then
  `agent kill --group <name>.<run_ts>` (daemon-performed). Agents spawned after a
  script unset `ORCR_GROUP` are deliberately outside this cleanup (logged).
- **pause** — no new fires; a recorded pending fire is held, not executed; active runs
  continue. **resume** — fires resume; a held pending fire executes if due.
  **rm** — mark ended (`removed` / `removed_by_run` when called from inside a run via
  `orcr loop rm $ORCR_LOOP`); no future fires; the active run and its agents continue
  unless `--kill-active`. Definition + run history remain queryable.
- `loop logs` = each run's captured stdout/stderr (`--run l3.r17` or a run timestamp).

### 4.3 daemon

```
orcr daemon start | stop | status
```

Auto-starts on demand (first `loop create`, first queued agent, first GC deadline).
Holds the store lease; owns loop firing, GC parking/reaping, queue promotion,
timed-out-run cleanup, and reconciliation (§6.5). `daemon status` is the health probe:
version, store path + `store_id`, herdr binary/server/session reachability + session
claim, live/queued/blocked counts, `foreign_in_owned_session` count, loop schedule and
next fires, reconciliation drift. (`herdr server` is the substrate process;
`orcr daemon` is the supervisor — two words, two processes.)

### 4.4 Foreign sessions — the exact contract

`--session <s>` switches an `agent` verb's target space to another herdr session.

| verb | with `--session <s>` |
| --- | --- |
| `run` | valid — the explicit escape hatch: spawns a **managed** agent (store row, GC, caps, group) whose tab is placed in that session's active workspace instead of the owned session. Never happens unasked. |
| `ls` | lists herdr-detected agents in `<s>` as **synthetic rows** (`origin: herdr`, no store row): herdr target id/label, detected harness, state, pane. Group filters don't apply (no metadata). |
| `send` / `attach` / `logs` | work on detected agents by herdr target; `logs` requires a reported `agent_session` compatible with an adapter (else `transcript_unsupported`). |
| `wait` | rides herdr agent states for detected agents (epoch discipline applies from orcr's own sends only). |
| `kill` | requires `--force` (orcr didn't start it). |

Detected foreign agents are never GC'd, queued, reconciled, or stored (beyond event
records of actions taken on them). Managed agents placed in a foreign session via
`run --session` remain fully managed (GC moves park them to the owned session's `idle`
workspace).

## 5 · SDK (TypeScript)

A thin typed wrapper over `orcr … --json` — no private protocol; anything the SDK does,
a shell script can do. Published as `@orchestratr/sdk` (name TBD). Python deferred.

```ts
import { orcr } from "@orchestratr/sdk";

// spawn — returns a handle immediately (agent run)
const a = await orcr.agent.run({
  harness: "codex",            // required
  prompt: "…",                 // -p   (or promptFile)
  name?, group?, gc?,          // --name --group --gc
  model?, effort?, cwd?, timeout?, session?,
});

a.id; a.name; a.group;         // resolved identity — group is the EFFECTIVE path
await a.wait({ timeout? });    // agent wait <id>
await a.send(text, { expect? });   // agent send [--expect]
await a.logs({ tail?, follow? });
await a.lastResponse();        // agent logs --last-response → string (throws TranscriptUnavailable)
await a.kill();

// collections
await orcr.agent.wait([a, b], { any?, timeout? });
await orcr.agent.wait({ group: "refactor.phase_1" }, { timeout? });
await orcr.agent.ls({ group?, all?, status? });
await orcr.agent.kill({ group: "refactor" });

// the one-liner: run + wait + lastResponse, gc immediate
const answer: string = await orcr.ask({ harness: "claude", prompt: "…" });

// grouping — SDK-local async context (AsyncLocalStorage), NOT process-global env:
// affects only orcr subprocesses spawned within fn's async call tree; composes with
// any inherited ORCR_GROUP; returns fn's result; exposes the effective group.
await orcr.withGroup("refactor", async (g /* effective path */) => { … });
// withGroup(path, { killOnThrow: true }) → agent kill --group <effective path> on throw;
// the rethrown error carries the cleanup result.

// durable scheduling — argv array (exec'd directly); string form runs via sh -c
await orcr.loop.create({ every: "30m", name: "burn_down",
                         command: ["npx", "tsx", "burn-down.ts"] });
await orcr.loop.rm(process.env.ORCR_LOOP!);   // a loop-run script ending its own loop
```

Errors: non-zero exits become typed errors carrying `{ code, message, details,
exitCode }` from the JSON envelope (`TranscriptUnavailable`, `StateConflict`, …).

**The file convention** (replaces v1's managed response files): when a step needs a
guaranteed-format answer, the prompt says where to write it — *"write your findings to
./out/security.md, then say DONE"* — and the script reads the file. Agents are good at
writing files; the CLI doesn't manage them. `ask()`/`lastResponse()` cover the casual
cases via transcripts.

Canonical patterns (classify-and-act, fanout-and-synthesize, adversarial verification,
generate-and-filter, tournament, loop-until-done + durable handoff) ship as documented
recipes in the SDK docs; all six are expressible with the surface above.

## 6 · Execution details

### 6.1 Spawn pipeline (`agent run`) — durable state before side effects

1. Load config; open store; discover herdr; resolve profile; resolve effective group
   (env prefix + `--group`, `/` = absolute); validate grammar/limits.
2. **One transaction**: allocate `a<N>`, insert the agent row with the full launch
   payload (prompt/file content, harness/model/effort, cwd, session, group, gc, timeout,
   env snapshot, `queue_seq`) and status `queued` or `starting` per admission (§3.4).
   The id is now durable — print it (async mode returns here for queued spawns too).
3. If starting: ensure session server (report `existing|started`); ensure the level-1
   workspace; `herdr agent start` in a new tab — profile argv, env contract
   (`ORCR_ID`, `ORCR_STORE_ID`, …), `--no-focus`. **Update the row with
   `workspace_id/tab_id/pane_id` immediately** after each herdr step.
4. Startup recipe; capture `agent_session_*` as soon as herdr reports it (a gate for
   `logs`; see §6.4).
5. Deliver the first message (input epoch 1; two-call rule).
6. If `--wait`: watch to completion (§3.6) — through queue promotion if queued — then
   print the final response or fail `transcript_unavailable`.

Crash safety: a `starting` row without a live pane past a bound → `failed` (reconciler);
a marked pane whose row lacks late fields → the row is repaired (adopted), never closed.

### 6.2 GC engine (daemon)

Tick ~30s, all transitions CAS-versioned: `gc auto` agents turn-complete + idle ≥
`idle_after` → pane moved to the `idle` workspace (`parked`, `parked_at` recorded);
parked ≥ `kill_after` → graceful kill (`exit_reason: reaped`). `gc immediate` agents:
two-phase — stable idle → transcript quiescence → final response recorded → kill; ends
`done` (`exit_reason: completed`). `send` un-parks inline (CLI): CAS `parked → live`,
cancel pending reap, move pane back, reset clocks, *then* deliver. No move/reap while
an attach is active (deferred + logged).

### 6.3 Loop scheduler (daemon)

Per loop: `next_fire_at` computed in the creating timezone (cron) or from run
completion/fire (`--every`), persisted as UTC. On fire: running-count <
`--concurrency` → start run `l<N>.r<seq>` in a fresh process group (pid/pgid recorded,
stdout/stderr captured); else overlap policy (`queue`: record `pending_due_at`,
coalescing; `skip`: log). On run exit: record status/exit code; if `pending_due_at` is
set and a slot is free, fire immediately. Per-run timeout: TERM `-pgid` → grace → KILL
`-pgid` → `agent kill --group <name>.<run_ts>`. Crash recovery: `running` rows are
verified against pgid existence on daemon start; dead ones are closed out and their
agents group-killed.

### 6.4 Transcript adapters & the capability matrix

Per-harness modules that locate and parse the harness's native session files into a
common shape (ordered messages, roles, tool calls, token counts). **Identity is a
gate, not a guess**: adapters select transcripts by the pane's `agent_session` id and
the agent's `created_at` — never by cwd mtime alone; multiple candidates → structured
error listing them. **Freshness**: a final response is only reported once the
transcript has advanced past the observed completion (bounded wait); otherwise
`transcript_unavailable`. Used by `logs`, `--last-response`, `run --wait`, `ask()`.

| harness | TUI orchestration | `logs` / `--wait` response |
| --- | --- | --- |
| claude | ✓ | ✓ (JSONL adapter, exists) |
| codex | ✓ | ✓ (JSONL adapter, exists) |
| pi | ✓ | ✗ until its adapter ships — `transcript_unsupported` (exit 2), stated in `--help` and the JSON result |
| opencode | ✓ | ✗ same |

### 6.5 Reconciliation

On daemon start and periodically: agents whose panes vanished → `lost`; panes in the
owned session carrying **this store's** marker but unknown/stale in the store → closed;
unmarked panes → counted and reported (`foreign_in_owned_session`), never closed;
in-flight `starting` markers are given their bound before any action. Queued agents
whose slots freed while no daemon ran → promoted (atomic CAS) by the next CLI
invocation.

### 6.6 Daemon-down matrix

| duty | while daemon is down |
| --- | --- |
| loop fires | not performed; skipped-and-logged on restart (never replayed) |
| pending fire execution | held; executes on restart if still due |
| queue promotion | inline by the next CLI invocation (atomic CAS + capacity recount) |
| GC park / reap | applied lazily: any CLI touch of an overdue agent applies the transition; daemon restart sweeps the rest |
| agent `--timeout` | enforced by a waiting caller's watcher if present; otherwise applied on next touch/restart (an unwatched agent may overrun while the daemon is down) |
| loop-run timeout | applied on daemon restart (pgid verified) |
| reconciliation | on daemon restart; `agent ls` marks rows whose pane probe fails as `lost` opportunistically |
| daemon auto-start fails | commands that require it (`loop create`) exit 2; stateless `agent` verbs proceed with a stderr warning |

## 9 · Configuration

# ~/.orcr/config.toml — every key optional; defaults shown
[defaults]
harness = "claude"

effort  = ""

bin     = ""            # empty = $ORCR_HERDR_BIN → $PATH

max    = 25             # global ceiling (RAM protection)
claude = 10             # per-harness caps beneath it (any harness name is a key)

idle_after = "5m"       # turn-complete + idle this long → parked in the idle workspace
kill_after = "10m"      # parked this long → reaped

§3.1 — a second store must use its own session); `ORCR_SESSION` overrides the session
per invocation; `ORCR_HERDR_BIN` overrides discovery.
