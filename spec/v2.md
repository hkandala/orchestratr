# orchestratr — design specification

orchestratr (CLI: `orcr`) is a cross-provider orchestrator for AI coding agents, built on
[herdr](https://herdr.dev). This document is the complete specification: problem,
architecture, core concepts, CLI, server API, SDK, execution model, storage, and
configuration. The implementation is built from scratch against this document.

Status: design locked pending final owner review; revision 4 (owner review pass).

---

## 1 · Problem statement

Coding agents (Claude Code, Codex CLI, Pi, OpenCode) are single-player tools: one
terminal, one session, one human watching. Real work wants many of them — a reviewer
fanned out per concern, a worker iterating under a verifier, a nightly job that triages
issues — often spanning *different* agent providers, since each has different strengths
and each subscription has its own quota.

Running agents as headless API calls solves scale but loses three things that matter:

1. **Plan pricing.** Interactive TUI sessions bill against flat subscription plans;
   API-mode invocations bill per token. At orchestration scale the difference is the
   whole budget.
2. **Steerability & visibility.** A headless agent can't be watched mid-flight, nudged
   with a correction, or taken over when it goes sideways. A real TUI can.
3. **Uniformity.** Every provider has a different headless mode, different flags,
   different transcripts. A script that fans out across providers needs one interface.

orcr's answer: run every agent as a **real interactive TUI in a herdr-managed terminal
pane**, and expose the smallest possible surface to spawn, message, await, read, and
stop them — from a shell, from any programming language, or from *inside another
agent*. herdr supplies the terminal substrate (sessions, panes, agent state detection,
attach-from-anywhere); orcr supplies identity, grouping, lifecycle, scheduling, and a
uniform cross-provider contract.

### Goals

- Agents as real TUIs: plan-pricing-safe, attachable, steerable, visible.
- One interface across agent providers, via per-provider **integrations** (claude and
  codex ship first; the integration surface is designed for more).
- Extreme-minimal primitives that compose from any language. **The server's socket API
  is the API** (mirroring herdr's own design); the CLI and the SDK are thin clients of
  it.
- Agents can orchestrate agents: any orcr-spawned agent can call `orcr` itself; lineage
  and grouping assemble automatically.
- Safe at scale: a single queue with global and per-provider concurrency caps (RAM
  protection), automatic lifecycle GC, one owned herdr session so the user's own
  workspace is never polluted.
- Organized at scale: hierarchical groups double as **identity** — every agent's unique
  id is its group path plus its name — and make a 200-agent workflow legible, operable
  (`wait`/`kill` by prefix), and visualizable both in herdr's native UI and in
  `orcr top`.
- Durable scheduling: run any command on a cadence, surviving the caller's shell.

### Non-goals (this version)

Cross-host orchestration from the local CLI (running orcr *on* a remote host over ssh
works today — see §6.7) · permission policies / read-only mode (providers run in their
bypass-permissions mode; every spawn result carries `"permissions": "bypass"`) ·
Windows · Python SDK · agent-to-agent coordination primitives (inboxes, decision gates,
task boards) · git worktree provisioning · pi / opencode integrations (surface is
specced; adapters land next).

## 2 · Architecture

```
you / a script / another agent
        │
        ├─ orcr CLI ────────────┐        (thin clients of the socket API)
        └─ TS SDK ──────────────┤
                                ▼
  orcr server  ── unix socket ~/.orcr/orcr.sock (JSON protocol, versioned, schema'd)
        │            owns: store (sqlite) · queue · GC · loops · reconcile · events
        ▼
  herdr (external binary + its own socket API — discovered, never embedded)
        └─ session "orcr" ─ workspaces (= level-1 groups) ─ tabs (= agents) ─ panes
                                └─ real TUIs: claude / codex / …
        └─ integrations read each provider's native transcript files
```

- **Server** — the single long-lived process and the **single writer**. Owns the store,
  the admission queue, GC, loop scheduling, reconciliation, and the event stream.
  Exposes everything over a Unix socket (§6.6) — the same shape as herdr itself, whose
  CLI subcommands are explicitly "helpers over the socket API". Auto-started on demand
  by any CLI/SDK call; `orcr server install` registers it to start at login (§4.4).
- **CLI** — every verb is a thin socket client; if the server isn't running it is
  auto-started first. If the server cannot start, commands exit 2 with the reason.
- **herdr driver** — the server shells the `herdr` CLI and parses its JSON envelopes.
  Discovery: config `herdr.bin` → `$ORCR_HERDR_BIN` → `$PATH`; missing → friendly
  pointer, exit 2.
- **Integrations** — one module per agent provider (named after herdr's own
  integrations): launch argv (bypass flags, model/effort mapping), startup recipe,
  completion-detection parameters, graceful-shutdown recipe, transcript adapter.
  **claude and codex ship built-in first**; the module boundary is the contract for
  adding more, and a future `orcr integration add|rm|ls` manages them like herdr does.
  Running a provider with no integration still works degraded: spawn/send/attach/kill
  and herdr-state-based wait all function; `logs`/`--last-response` return
  `integration_missing`, and completion detection falls back to generic idle-stability
  defaults.
- **Store** — sqlite (WAL) under `~/.orcr/`, owned exclusively by the server. Schema
  in §7.

## 3 · Core concepts

### 3.1 Identity: group + name (no other ids)

Every agent's unique identifier is **its group path plus its name**, joined with dots:

```
<group>.<name>        default.k3f9x · refactor.phase_1.file_1
```

- **Name** — one segment, `[a-z0-9_]+`, ≤ 64 chars. User-provided via `--name`, or
  auto-generated as a short 5-char lowercase alphanumeric sequence (e.g. `k3f9x`,
  regenerated on collision). No UUIDs, no numeric counters anywhere in the surface.
- **Group** — dot-separated segments, each `[a-z0-9_]+`; ≤ 6 segments, ≤ 64
  chars/segment, ≤ 256 chars total (name included). Violations are rejected at spawn
  with a clear error, never truncated. Default group: `default`.
- **Uniqueness** — `(group, name)` must be unique among **live + queued** agents. The
  same name may exist in different groups (`review.abc` and `verify.abc` coexist).
  Names of ended agents are reusable; a bare id reference resolves to the live/queued
  agent first, else the most recent ended agent with that id (for `logs` on history).
- `agent run` prints the **full id** (`refactor.phase_1.file_1`) on stdout.
- **Prefix operations** — `ls`, `wait`, `kill`, and the loop verbs take an **id
  prefix**, matched on segment boundaries (`phase_1` never matches `phase_10`; a
  prefix matches ids equal to it or extending it past a dot). A full id therefore
  matches exactly one agent; a group path matches its subtree. Verbs that need exactly
  one target (`send`, `logs`, `attach`) require the prefix to resolve to a single
  agent — multiple matches are an error listing the candidates.
- **Same name = same group, by definition.** No auto-suffixing. Scripts wanting
  per-run isolation stamp their own suffix (`--group "refactor.$(date +%s)"`).

**Display transform** (derives human labels from machine slugs; no stored metadata):
underscores → spaces, words title-cased, dots → " / ".
`phase_1.review.file_1` → **Phase 1 / Review / File 1**. Every result and `ls` row
carries the machine id alongside the display form; TTY output always shows the machine
id so prefixes can be copied, not guessed.

**Inheritance — relative by default.** *Effective group = inherited prefix `.`
explicit `--group`* (prefix alone if no flag; flag alone if no prefix; `default` if
neither). The inherited prefix comes from the calling context's `ORCR_ID` (§3.3): for
an agent, its group (id minus name); for a loop run, the full run path
`<loop_name>.<run_ts>` (§4.3). A **leading `/` makes `--group` absolute** — it ignores
any inherited prefix (`/` is outside the segment charset, so this is unambiguous).
Plain scripts outside any agent compose their own prefixes explicitly (or via SDK
`orcr.group()`, §5).

### 3.2 The owned session & the herdr mapping

All orcr-managed agents live in one dedicated herdr session (default name `orcr`,
config `herdr.session`). The user's daily herdr session never sees a subagent pane.

- First use auto-starts the session's herdr server headless; `herdr --session orcr`
  opens the native UI over it.
- `orcr agent attach` wraps `herdr agent attach`, which streams a pane's terminal into
  the current terminal from anywhere — no session switching.
- **Ownership marker**: every pane orcr creates carries `ORCR_ID` in its env and has a
  matching store row. Reconciliation (§6.5) acts only on panes that carry the marker
  *and* match a store row; unmarked panes in the owned session (a shell you opened
  while debugging) are reported in `server status` and never touched automatically.

Within the session, herdr's hierarchy is used as follows. (herdr facts: workspaces are
per-session, purely visual containers with a label and no cwd; only panes have a cwd
and a process.)

| herdr level | orcr's use |
| --- | --- |
| workspace | = the agent's **level-1 group segment**: everything under `refactor.*` → workspace `refactor`; agents with no group → workspace `default`; each loop → one workspace named after the loop. GC-parked agents → workspace `idle`. |
| tab | one per agent; label = **remaining group path + name**: id `refactor.phase_1.review.xyz123` → workspace `refactor`, tab `phase_1.review.xyz123`. |
| pane | the agent's TUI; cwd = caller's cwd or `--cwd`. A `pane_id` is stable for the *pane*, but **not an agent identifier**: GC moves agents across panes/workspaces over their lifetime (verified: `herdr pane move --new-tab --workspace <id>` moves panes across workspaces). The store tracks the agent's *current* pane as a location column, nothing more. |

herdr removes a workspace automatically once it has no panes — so orcr always **closes
panes** it is done with (kill, reap, gc-immediate); closing the last pane closes the
tab, and emptying the workspace removes it. Leaving a stray pane behind would pin the
workspace forever.

### 3.3 Env contract

Exactly two variables, injected into every managed agent pane and every loop-run
command:

```
ORCR_ID         the agent's full id (group.name) — or, in a loop-run command,
                the run path <loop_name>.<run_ts>
ORCR_PARENT_ID  the ORCR_ID of the context that spawned this agent
                (an agent or a loop run); unset for agents started at the root
```

Everything else is derived: group = `ORCR_ID` minus the name segment, loop and run
from the run path. When `orcr agent run` executes inside a managed context, the server
sets the child's `ORCR_PARENT_ID` to the caller's `ORCR_ID` and computes the effective
group from it (§3.1). Parent lineage is what `orcr top` (§4.2) draws.

### 3.4 Lifecycle (GC)

One policy: `--gc auto|immediate|never`. `--gc` governs **pane lifetime only** —
history in the store is unaffected. GC applies only to **managed** agents (§3.7).

| mode | behavior |
| --- | --- |
| `auto` (default) | turn-complete and idle for `idle_after` (5m) → pane moved to the `idle` workspace (`parked`) → `kill_after` (10m) more → graceful kill, memory released. An inbound `send` at any point moves the agent back to its workspace and resets both clocks. |
| `immediate` | pane closed as soon as the first turn completes **and its final response has been captured** (two-phase: stable idle → transcript quiescence → response recorded → kill). The agent ends `done` (`exit_reason: completed`), not `killed` — `run --wait` on it exits 0. |
| `never` | exempt from parking and reaping; lives until an explicit `agent kill`. For pinned long-livers (heartbeat agents). |

**There is no default timeout.** An agent never times out unless the caller passed an
explicit `--timeout <dur>` (then: kill with `exit_reason: timeout` on expiry). A
stuck-working agent otherwise stays alive and visible (`blocked`/`working` in `ls` and
`top`) until a human or script acts.

**Interlocks** (all state transitions are versioned compare-and-swap on the agent
row): `send` cancels a pending park/reap atomically *before* delivering input; GC
never moves or reaps a pane with an active attach; completion capture and GC-kill are
ordered (the response is recorded before the pane dies).

**Known caveat — background subagents (documented, not solved in this version).**
Claude Code sometimes reports its main turn idle while background subagents are still
running; herdr then reports `idle`. Under `gc auto` the agent may be parked; when the
subagents return (typically ≤ 15m) it goes `working` again and is un-parked back to
its original workspace, so work is not lost — but a `kill_after` shorter than the
subagents' runtime could reap it mid-flight. Detecting in-flight background subagents
(via the transcript) is a planned integration capability; until then, use
`--gc never` for agents known to fan out background work.

### 3.5 Queue & concurrency

**Every `agent run` enqueues.** The verb's job is to validate, persist, and print the
id; the **server** processes the queue and manages the whole lifecycle. Every agent
therefore passes through the same states: `queued → starting → live (working → idle)
→ … → ended`.

- **Global cap** `concurrency.max` (default 25) — RAM protection; heavy TUIs at 100×
  will take a machine down.
- **Per-provider caps** beneath it (e.g. `claude = 10`); promotion needs a free slot
  in both.
- Promotion is strictly FIFO by `queue_seq`, as an atomic store transaction
  (`queued → starting` only if the row is still queued *and* a capacity recount under
  the write lock shows free slots). A `starting` row that makes no progress within a
  bound is reconciled to `failed`.
- `wait` on a queued agent waits through promotion; `kill` on a queued agent dequeues
  it (`exit_reason: canceled`).
- `--pane` placements (§3.7) bypass the queue and don't count toward caps — the caller
  owns that pane.
- Loops have a separate, unrelated knob: `--concurrency` caps concurrent *runs of that
  loop* (§4.3).

### 3.6 Completion discipline (internal, hard-won)

herdr reports agent state (`working / idle / blocked / unknown`). Completion is
defined per **input epoch**: every delivered input (the first prompt, every `send`)
increments the agent's `input_seq` *before* delivery. A turn is complete when, for the
latest epoch: `working` has been observed **after that epoch's delivery began** (small
per-integration grace window for fast turns), followed by **stable idle** — stability
defined per integration as a minimum idle duration *and* transcript quiescence. A
first idle without epoch-scoped working is never completion; an old idle can never
satisfy a newer send. `blocked` is epoch-scoped too (categories best-effort:
`question|limit|login|unknown`): it surfaces as status + exit 4 for waiting callers
and is clearable by `send`.

Other herdr driver rules: input delivery is two calls (send-text → ~1s → enter —
never one); herdr `--timeout` values are milliseconds and never leak into orcr's user
surface.

### 3.7 Managed vs unmanaged agents

orcr tracks **all** agents herdr can see — including ones the user started by hand in
their own sessions — but only *manages* the ones it created through the default flow.

- **Managed** — created by `agent run` in the owned session. Full lifecycle: queue,
  caps, GC, group/name, parent lineage, env contract.
- **Placed** (`agent run --pane <pane_id>`) — orcr kills whatever process is running
  on that pane and starts the agent there. It gets a store row, id, env contract, and
  all verbs work — but **no GC, no parking, no queue/caps**: the caller owns the
  pane's lifetime and placement. `run` never targets a pane in someone else's terms —
  `--pane` is the only way run touches a pane orcr didn't allocate.
- **Unmanaged (detected)** — agents herdr detects in the user's own sessions. The
  server tracks them in the store while it runs: provider, state
  (working/idle/blocked), active/closed, and the reported `agent_session` (which makes
  `logs` work when an integration exists). Identity is auto-assigned: group
  `unmanaged`, name = the herdr pane id sanitized to the name charset (e.g.
  `unmanaged.w6_p1`) — users cannot name or group them. Limitations, explicitly:
  no GC/parking · no queue or caps · no parent lineage (`ORCR_PARENT_ID` unknowable) ·
  no custom name/group · `run` is invalid in unmanaged sessions (their definition is
  that orcr didn't create them) · `kill` requires `--force` (it closes a pane orcr
  doesn't own). `send`, `attach`, `logs`, `wait`, `ls`, and `top` all work.

## 4 · CLI

Four nouns (`agent`, `loop`, `server`, `api`) plus `orcr top`. **Every command supports
`--json`** (exactly one envelope object on stdout — `{"ok":true,"result":…}` /
`{"ok":false,"error":{code,message,details}}` — logs to stderr). Exit codes: `0` ok ·
`2` environment · `3` timeout · `4` blocked · `5` killed/ended · `6` not found · `7`
state conflict · `1` other. Durations always carry units (`45s`, `20m`, `3h`).

### 4.1 agent

```
orcr agent run    -a <provider> [-p <prompt>]
                  [--name <n>] [--group <path>] [--gc auto|immediate|never]
                  [--model <m>] [--effort <e>] [--cwd <dir>] [--timeout <dur>]
                  [--pane <pane_id>] [--wait] [--json]
orcr agent send   <id> <prompt> [--json]
orcr agent logs   <id> [--last-response] [--tail <n>] [--follow] [--json]
orcr agent wait   <id-prefix>... [--status idle|working|blocked|ended]
                  [--timeout <dur>] [--json]
orcr agent attach <id> [--takeover]
orcr agent kill   <id-prefix>... [--force] [--json]
orcr agent ls     [<id-prefix>] [--agent <provider>] [--status <s>]
                  [--managed|--unmanaged] [--all] [--json]
```

**run** — **async: validates, enqueues, prints the full id (`group.name`; stdout
carries only the id in non-JSON mode) and returns**; a TTY also gets a stderr hint
(`wait: orcr agent wait refactor.k3f9x · response: orcr agent logs refactor.k3f9x
--last-response · attach: orcr agent attach refactor.k3f9x`). `--wait` blocks through
the queue and the first completion and prints the final response from the transcript;
if the turn completed but no final response can be identified, it exits 1 with
`error.code: transcript_unavailable`. Placement per §3.2 (or `--pane`, §3.7),
admission per §3.5, identity per §3.1, gc per §3.4. Prompts are plain text; if a step
needs files attached or a guaranteed-format answer, say so in the prompt (agents read
paths you mention and write files where you tell them — §5's file convention).

**send** — types the prompt into the agent's TUI and submits, whatever state the agent
is in (provider TUIs queue mid-turn input natively). It waits for the delivery to be
confirmed on the pane and returns success or failure — the result reports
`delivered_while: working|idle|parked` + `input_seq`. Sending to a parked agent
un-parks it (atomically, before delivery). Ended target → exit 7; unknown → exit 6.
*Planned (documented, next version): per-provider steer/stop options — interrupting an
active turn or gracefully stopping the current task differs per provider.*

**logs** — reads the provider's **native transcript** via the integration's adapter
(structured turns, tool calls, token counts where available). `--last-response` prints
only the final assistant message and **fails loudly rather than guessing**: exit 1
`transcript_unavailable` when no final response is identifiable; exit 2
`integration_missing` when the provider has no integration (§6.4). `--follow` streams.
Works on unmanaged agents when herdr reports a compatible `agent_session`; ambiguous
transcript candidates are an error listing the candidates, never a silent pick.

**wait** — targets are id prefixes; membership = live + queued agents matching any
prefix, **snapshotted at invocation**. `--status` (default `idle`) is the state being
waited for: `idle` = the latest epoch's turn-complete (an already-complete idle agent
returns immediately); `working` = the agent has started working (useful right after
`send`); `blocked` = needs a human; `ended` = fully gone. Returns when **all**
targets reach the status. Exits: `0` all reached · `4` a target blocked while waiting
for idle · `5` a target ended before reaching the status · `3` `--timeout` expired ·
`6` no target matched. JSON reports every target as `{id, state, turn, reached}`.

**attach** — wraps `herdr agent attach`: streams the pane into the current terminal
from anywhere (inside or outside herdr), detach returns. Observe by default,
`--takeover` claims input.

**kill** — graceful per-integration shutdown recipe (~5s deadline) → **pane closed**
(so herdr can clear empty tabs/workspaces); status ends `killed`; history remains.
Prefixes make subtree cleanup one command; queued agents are dequeued (`canceled`).
Partial success exits 0 with `killed[]` and `skipped[]` arrays. Unmanaged targets
require `--force`.

**ls** — live + queued agents (managed and unmanaged) rendered as the group tree;
headings show the display label *and* the machine id. TTY columns:
`ID STATUS AGENT GROUP AGE`. Filters: an id prefix (a full id lists exactly that
agent), `--agent <provider>`, `--status blocked` (= who needs a human),
`--managed`/`--unmanaged`, `--all` (include ended agents — history, including every
past loop run). JSON rows are flat: `{id, name, group, group_display, state, turn,
agent, managed, cwd, pane_id, queue_position?, parent_id?, created_at, ended_at?,
exit_reason?}`.

### 4.2 top

```
orcr top [<id-prefix>] [--agent <provider>] [--status <s>] [--loops]
```

The live dashboard: a realtime TUI tree of all agents — grouped by their group
hierarchy, parent→child edges drawn from `ORCR_PARENT_ID` lineage, states updating
live (queued/working/idle/blocked/parked), loops and their active runs shown as
subtrees (`--loops` filters to loops only). Same filters as `ls`. This is the default
view for tracking a running workflow or loop.

*Planned (documented, next version): per-agent live activity in the tree — tool calls
and response summaries streamed from the transcripts.*

### 4.3 loop

```
orcr loop create ("<cron>" | --once-at <time>)
                 [--name <n>] [--concurrency <n>] [--overlap queue|skip]
                 [--timeout <dur>] [--json] -- <command…>
orcr loop run    <name-prefix> [--json]
orcr loop ls     [<name-prefix>] [--status <s>] [--all] [--json]
orcr loop logs   <name-prefix> [--source orcr|command] [--tail <n>] [--follow] [--json]
orcr loop pause  <name-prefix> [--json]
orcr loop resume <name-prefix> [--json]
orcr loop rm     <name-prefix> [--kill-active] [--json]
```

Durable cron for **any command** — the `--` boundary captures an **argv array**,
executed directly (no shell). Want shell features? Say so: `-- sh -c 'a && b'`.
Creation echoes the parsed argv verbatim, the cadence in words (local + UTC), and the
exact cancel command. The command spawns agents via CLI/SDK like any other caller; the
loop owns *time only*: no provider flags, no prompts, no judge logic, no stop-condition
DSL.

- **Name = group.** A loop's name is one group segment (`[a-z0-9_]+`); default
  `loop_<5-char alnum>`. The loop gets its own workspace (level-1 group = its name).
- **Cadence**: five-field cron — stored **with the creating timezone** and evaluated
  in it (DST-correct: "9am weekdays" stays 9am), each occurrence persisted as a UTC
  `next_fire_at` · or `--once-at <time>` (fires once then ends). There is no
  `--every` — intervals are cron expressions (`*/30 * * * *`). Fires missed while the
  machine slept or the server was down are skipped and logged, never replayed.
- **`loop run`** — queues an immediate run (subject to the loop's concurrency/overlap
  policy). Works on paused loops too — it's the manual trigger.
- **Runs & grouping**: each run's identity is its path **`<loop_name>.<run_ts>`** —
  `run_ts` is the fire's unix seconds, bumped to the next unused second on collision.
  The run command executes in its **own process group** (pid/pgid recorded) with
  `ORCR_ID=<loop_name>.<run_ts>`, so every agent it spawns lands under that path:
  a script's `--group review` + `--name file_1` yields
  `nightly.1783206000.review.file_1`; agents it doesn't name get auto names.
  `orcr agent ls --all nightly` is the loop's full agent history.
- **Concurrency & overlap**: `--concurrency` caps concurrent runs (default 1). At the
  cap, `--overlap queue` (default) records **at most one pending fire** (later fires
  coalesce into it; survives server restarts); `skip` drops the fire with a log line.
  When a run exits and a pending fire is recorded, it fires immediately.
- **Per-run timeout** (only if `--timeout` was given — no default): TERM to the run's
  process group → grace → KILL → then `orcr agent kill <name>.<run_ts>`
  (server-performed).
- **pause** — no new fires; a recorded pending fire is held, not executed; active runs
  continue. **resume** — fires resume; a held pending fire executes if due. **rm** —
  mark ended (`removed` / `removed_by_run` when called from inside a run via
  `orcr loop rm $ORCR_LOOP_NAME` — i.e. the first segment of `$ORCR_ID`); no future
  fires; the active run and its agents continue unless `--kill-active`. Definition +
  run history remain queryable. All verbs accept **name prefixes** to operate on many
  loops at once (`orcr loop pause nightly` pauses `nightly`; `orcr loop pause night`
  matches nothing — segment-boundary rule).
- **`loop logs`** — two interleaved sources, each line tagged with its run
  (`[nightly.1783206000]`): the **command's** captured stdout/stderr, and **orcr's own
  actions** on the loop (fired, coalesced, skipped, timed out, killed — from the event
  log). `--source orcr` / `--source command` filters to one side.

### 4.4 server

```
orcr server start | stop | status | install | uninstall
```

The orcr server (§2): single writer, queue, GC, loops, reconciliation, socket API.
Auto-started on demand by any CLI/SDK call; `install` registers it to start at login /
system start (launchd on macOS; systemd user unit on Linux), `uninstall` removes the
registration. `server status` is the health probe: version, protocol version, socket
path, store path, herdr binary/server/session reachability, live/queued/blocked
counts, unmarked-panes-in-owned-session count, loop schedule and next fires,
reconciliation drift.

### 4.5 api

```
orcr api schema [--json | --output <path>]
orcr api snapshot [--json]
```

Mirrors `herdr api`: `schema` prints the versioned JSON schema of the socket protocol
(requests, responses, events); `snapshot` dumps live runtime state (agents, queue,
loops, GC clocks) in one document. These make the socket API self-describing for
non-TS languages — the schema is the contract, the CLI is one client of it.

## 5 · SDK (TypeScript)

A typed client of the **socket API** (§6.6) — same protocol as the CLI, no private
surface; anything the SDK does, a shell script can do with `orcr … --json`. Published
as `@orchestratr/sdk` (name TBD). Python deferred.

```ts
import { orcr } from "@orchestratr/sdk";

// spawn — returns a handle immediately (agent run semantics)
const a = await orcr.agent.run({
  agent: "codex",              // required — the provider
  prompt: "…",
  name?, group?, gc?,          // --name --group --gc
  model?, effort?, cwd?, timeout?, pane?,
});

a.id;                          // full id: "refactor.phase_1.k3f9x"
a.name; a.group;
await a.wait({ status?, timeout? });   // agent wait — status defaults to "idle"
await a.send(prompt);                  // agent send
await a.logs({ tail?, follow? });
await a.lastResponse();        // logs --last-response → string (throws TranscriptUnavailable)
await a.kill();

// prefix collections — same prefix semantics as the CLI
await orcr.agent.wait("refactor.phase_1", { status?, timeout? });
await orcr.agent.ls({ prefix?, agent?, status?, managed?, all? });
await orcr.agent.kill("refactor");

// the one-liner: run + wait + lastResponse, gc immediate
const answer: string = await orcr.ask({ agent: "claude", prompt: "…" });

// grouping — async-context scoped (AsyncLocalStorage), NOT process-global:
// every orcr call inside fn's async call tree composes under the prefix;
// nests with any inherited context; returns fn's result.
await orcr.group("refactor", async (g /* effective prefix */) => { … });
// orcr.group(path, { killOnThrow: true }) → orcr.agent.kill(<prefix>) on throw

// live events — the socket's subscription stream (what `orcr top` renders)
const sub = await orcr.watch({ prefix? });   // AsyncIterable<AgentEvent>
for await (const ev of sub) { /* state changes, queue moves, loop fires */ }

// durable scheduling — argv array (exec'd directly); string form runs via sh -c
await orcr.loop.create({ cron: "*/30 * * * *", name: "burn_down",
                         command: ["npx", "tsx", "burn-down.ts"] });
await orcr.loop.run("burn_down");            // manual trigger
await orcr.loop.rm(loopNameFrom(process.env.ORCR_ID!));  // self-terminate from a run
```

Errors: failures become typed errors carrying `{ code, message, details }` from the
protocol (`TranscriptUnavailable`, `IntegrationMissing`, `StateConflict`, …).

**The file convention**: when a step needs a guaranteed-format answer, the prompt says
where to write it — *"write your findings to ./out/security.md, then say DONE"* — and
the script reads the file. Agents are good at writing files; orcr doesn't manage
prompt/response files at all. `ask()`/`lastResponse()` cover the casual cases via
transcripts.

Canonical patterns (classify-and-act, fanout-and-synthesize, adversarial verification,
generate-and-filter, tournament, loop-until-done + durable handoff) ship as documented
recipes in the SDK docs; all six are expressible with the surface above.

## 6 · Execution details

### 6.1 Spawn pipeline (`agent run`) — durable state before side effects

1. CLI/SDK sends the run request over the socket (auto-starting the server if
   needed). The server: loads config, discovers herdr, resolves the integration,
   resolves the effective group (inherited prefix from the caller's `ORCR_ID` +
   `--group`, `/` = absolute), allocates or validates the name; validates
   grammar/limits and `(group, name)` uniqueness.
2. **One transaction**: insert the agent row with the full launch payload
   (prompt, provider/model/effort, cwd, group, name, gc, timeout, env snapshot,
   `queue_seq`) and state `queued`. The id is now durable — the verb returns it
   (async mode ends here for the caller).
3. Queue promotion (§3.5) picks it up: ensure the owned session's herdr server;
   ensure the level-1 workspace; `herdr agent start` in a new tab — integration argv,
   env contract (`ORCR_ID`, `ORCR_PARENT_ID`), `--no-focus`. **The row is updated
   with `workspace_id/tab_id/pane_id` immediately** after each herdr step.
4. Startup recipe; capture `agent_session_*` as soon as herdr reports it (the gate
   for `logs`; §6.4).
5. Deliver the first prompt (input epoch 1; two-call rule).
6. `--wait` callers hold a subscription on the agent's events through promotion and
   first completion, then fetch the final response (or fail
   `transcript_unavailable`).

`--pane <pane_id>` runs replace steps 3's placement: the server kills the pane's
current process, starts the integration argv there, and skips queue/caps/GC (§3.7).

Crash safety: a `starting` row without a live pane past a bound → `failed`
(reconciler); a marked pane whose row lacks late fields → the row is repaired
(adopted), never closed.

### 6.2 GC engine (server)

Tick ~30s, all transitions CAS-versioned: `gc auto` agents turn-complete + idle ≥
`idle_after` → pane moved to the `idle` workspace (`parked`, `parked_at` recorded);
parked ≥ `kill_after` → graceful kill (`exit_reason: reaped`) and **pane closed**.
`gc immediate` agents: two-phase — stable idle → transcript quiescence → final
response recorded → kill + pane closed; ends `done` (`exit_reason: completed`).
`send` un-parks: CAS `parked → live`, cancel pending reap, move pane back, reset
clocks, *then* deliver. No move/reap while an attach is active (deferred + logged).
Placed (`--pane`) and unmanaged agents are never GC'd.

### 6.3 Loop scheduler (server)

Per loop: `next_fire_at` computed in the creating timezone, persisted as UTC. On fire
(or `loop run`): running-count < `--concurrency` → start the run
`<loop_name>.<run_ts>` in a fresh process group (pid/pgid recorded, stdout/stderr
captured to the run's log); else overlap policy (`queue`: record `pending_due_at`,
coalescing; `skip`: log). On run exit: record status/exit code; if a pending fire is
recorded and a slot is free, fire immediately. Per-run timeout (explicit only): TERM
`-pgid` → grace → KILL `-pgid` → `agent kill <name>.<run_ts>`. Every scheduler action
(fired, coalesced, skipped, paused-hold, timeout, kill) is an event row — that's
`loop logs --source orcr`. Crash recovery: `running` rows are verified against pgid
existence on server start; dead ones are closed out and their agents prefix-killed.

### 6.4 Integrations & the capability matrix

One module per provider: launch argv (bypass-permissions flags, model/effort mapping),
startup recipe, completion-detection parameters (idle stability, grace windows),
graceful-shutdown recipe, and the **transcript adapter** — locating and parsing the
provider's native session files into a common shape (ordered messages, roles, tool
calls, token counts). **Identity is a gate, not a guess**: adapters select transcripts
by the pane's `agent_session` id and the agent's `created_at` — never by cwd mtime
alone; multiple candidates → structured error listing them. **Freshness**: a final
response is only reported once the transcript has advanced past the observed
completion (bounded wait); otherwise `transcript_unavailable`.

| provider | TUI orchestration | `logs` / `--wait` response | status |
| --- | --- | --- | --- |
| claude | ✓ | ✓ | built-in, first release |
| codex | ✓ | ✓ | built-in, first release |
| pi / opencode / … | ✓ degraded (generic completion defaults) | ✗ `integration_missing` (exit 2), stated in `--help` and JSON | next; `orcr integration add` planned |

### 6.5 Reconciliation & unmanaged discovery

On server start and periodically: managed agents whose panes vanished → `lost`; panes
in the owned session carrying an `ORCR_ID` marker but unknown/stale in the store →
closed; unmarked panes in the owned session → counted and reported in
`server status`, never closed. In the user's other sessions, herdr-detected agents are
**discovered into the store** as unmanaged rows (`unmanaged.<pane>`, §3.7) and kept
current (state changes, closure) while the server runs.

### 6.6 The socket API

- **Transport**: Unix domain socket at `~/.orcr/orcr.sock` (mode 0600) — the same
  approach as herdr (`~/.config/herdr/herdr.sock`), which is why there's no TCP port:
  filesystem permissions are the auth story. A TCP/HTTP listener is a possible later
  addition for remote tooling, off by default.
- **Protocol**: JSON request/response plus **subscription events** (agent state
  changes, queue movements, loop fires) — versioned with a protocol number and
  described by a published JSON schema (`orcr api schema`), exactly like herdr's
  protocol/schema surface. `orcr api snapshot` dumps live state.
- **Clients**: the CLI (every verb), the SDK (including `orcr.watch()`), `orcr top`
  (renders the subscription stream), and anything else that can speak
  newline-delimited JSON over a socket.

### 6.7 Remote hosts (documented; not built)

herdr's remote story is per-host: `herdr --remote <ssh-target>` attaches your terminal
to a herdr *server running on the remote machine* — there is no cross-host pane
management. orcr mirrors that shape: the orcr server talks to the herdr socket on
**its own host**. Consequently, orchestrating agents on a remote machine works today
by running orcr *on that machine* (over ssh) — the entire lifecycle (queue, GC,
loops, transcripts) is host-local and needs zero changes. What is **not** built:
driving a remote host from a local `orcr` CLI (it would need the socket tunneled,
remote transcript access, and remote process-group control for loops). Parked.

## 7 · Store

sqlite, WAL, under `~/.orcr/`, owned exclusively by the server (single writer). Built
from scratch for this design — there is no prior schema to migrate.

```
agents:    rowid PK,
           id (group_path || '.' || name — unique among live+queued),
           group_path, name, managed (0|1), origin (run|pane|detected),
           parent_id,                                  -- ORCR_ID of spawning context
           agent (provider), model, effort, gc_mode, cwd,
           workspace_id, tab_id, pane_id,              -- current location, not identity
           agent_session_kind, agent_session_value,    -- transcript identity gate
           state (queued|starting|live|parked|ended|lost),
           turn  (none|working|blocked|complete),
           blocked_kind (question|limit|login|unknown),
           input_seq, exit_reason (completed|killed|canceled|reaped|timeout|failed),
           launch_json,                                -- full relaunch payload
           queue_seq, enqueued_at, deadline_at,        -- deadline only if --timeout given
           idle_since, parked_at, last_state_change_at, created_at, ended_at
loops:     name PK (unique among active), cadence_kind (cron|once), cadence_value, tz,
           command_json (argv), concurrency, overlap, timeout_s (nullable),
           status (active|paused|ended), next_fire_at, last_fire_at,
           pending_due_at, created_at, ended_reason (removed|removed_by_run|fired)
loop_runs: loop_name, run_ts (PK pair; the run's group path suffix),
           pid, pgid, status (running|ok|failed|timeout), exit_code,
           log_path, started_at, ended_at
events:    seq PK AUTOINCREMENT, ts, kind, ref_id, payload_json
           -- append-only audit; also the source for `loop logs --source orcr`
```

Prefix queries are indexed scans on `id`, matched on `.` boundaries.

## 9 · Configuration

```jsonc
// ~/.orcr/config.json — every key optional; defaults shown
{
  "defaults": {
    "agent": "claude",        // default provider
    "model": "",              // empty = provider default
    "effort": ""
    // no default timeout — agents never time out unless --timeout is passed
  },
  "herdr": {
    "bin": "",                // empty = $ORCR_HERDR_BIN → $PATH
    "session": "orcr"         // the owned session; user sessions are never touched
  },
  "concurrency": {
    "max": 25,                // global ceiling (RAM protection)
    "claude": 10              // per-provider caps beneath it (any provider is a key)
  },
  "lifecycle": {
    "idle_after": "5m",       // turn-complete + idle this long → parked
    "kill_after": "10m"       // parked this long → reaped
  }
}
```

Env: `ORCR_HOME` relocates `~/.orcr` (store, socket, config — tests/sandboxes; pair it
with a distinct `herdr.session`); `ORCR_HERDR_BIN` overrides herdr discovery.
