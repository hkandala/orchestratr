# orchestratr v2 — design specification

orchestratr (CLI: `orcr`) is a cross-harness orchestrator for AI coding agents, built on
[herdr](https://herdr.dev). This document is the complete v2 specification: problem, core
concepts, CLI, SDK, execution model, storage, and configuration. It supersedes the v1
surface described in `spec/01–10` (the execution machinery there — herdr driver rules,
profiles, store discipline — carries forward except where this document says otherwise).

Status: design locked pending final owner review; not yet implemented.

---

## 1 · Problem statement

Coding agents (Claude Code, Codex CLI, Pi, OpenCode) are single-player tools: one
terminal, one session, one human watching. Real work wants many of them — a reviewer
fanned out per concern, a worker iterating under a verifier, a nightly job that triages
issues — often spanning *different* harnesses, since each has different strengths and
each subscription has its own quota.

Running agents as headless API calls solves scale but loses three things that matter:

1. **Plan pricing.** Interactive TUI sessions bill against flat subscription plans;
   API-mode invocations bill per token. At orchestration scale the difference is the
   whole budget.
2. **Steerability & visibility.** A headless agent can't be watched mid-flight, nudged
   with a correction, or taken over when it goes sideways. A real TUI can.
3. **Uniformity.** Every harness has a different headless mode, different flags,
   different transcripts. A script that fans out across harnesses needs one interface.

orcr's answer: run every agent as a **real interactive TUI in a herdr-managed terminal
pane**, and expose the smallest possible command surface to spawn, message, await, read,
and stop them — from a shell, from any programming language, or from *inside another
agent*. herdr supplies the terminal substrate (sessions, panes, agent state detection,
attach-from-anywhere); orcr supplies identity, grouping, lifecycle, scheduling, and a
uniform cross-harness contract.

### Goals

- Agents as real TUIs: plan-pricing-safe, attachable, steerable, visible.
- One interface across harnesses (claude / codex / pi / opencode; extensible per-harness
  profiles).
- Extreme-minimal primitives that compose from any language — the CLI is the API; the
  SDK is a convenience, not a requirement.
- Agents can orchestrate agents: any orcr-spawned agent can call `orcr` itself; lineage
  and grouping assemble automatically.
- Safe at scale: global and per-harness concurrency caps (RAM protection), automatic
  lifecycle GC, one owned herdr session so the user's own workspace is never polluted.
- Organized at scale: hierarchical groups make a 200-agent workflow legible, operable
  (`wait`/`kill` by group), and visualizable in herdr's native UI.
- Durable scheduling: run any command on a cadence, surviving the caller's shell.

### Non-goals (v2)

Remote hosts · permission policies / read-only mode (v2 runs harnesses in their
bypass-permissions mode; every spawn result carries `"permissions": "bypass"`) ·
Windows · Python SDK · agent-to-agent coordination primitives (inboxes, decision gates,
task boards) · git worktree provisioning · a monitoring TUI (`herdr --session orcr` *is*
the dashboard).

## 2 · Architecture

```
you / a script / another agent
        │  (shell, or TS SDK over --json)
        ▼
  orcr CLI  ──► store (sqlite: agents · loops · runs · events)
        │              ▲
        │              │ single-writer for durable state
        ▼              │
  orcr daemon ─────────┘   loops · GC mover/reaper · queue promotion · reconcile
        │
        ▼
  herdr (external binary, discovered — never embedded)
        └─ session "orcr" ─ workspaces (= level-1 groups) ─ tabs (= agents) ─ panes
                                └─ real TUIs: claude / codex / pi / opencode
        └─ transcript adapters read each harness's native session files
```

- **CLI** — all verbs; stateless ones (`agent *`) work with no daemon.
- **Daemon** — auto-started on demand; owns everything time-based: loop firing, GC
  parking/reaping, FIFO queue promotion, reconciliation. Single writer for durable
  state; foreground verbs coordinate through it when it's up, a store lock when not.
- **herdr driver** — shells the `herdr` CLI, parses its JSON envelopes. Discovery:
  config `herdr.bin` → `$ORCR_HERDR_BIN` → `$PATH`; missing → friendly pointer, exit 2.
- **Profiles** — one module per harness: launch argv (bypass flags, model/effort
  mapping), startup recipe, completion detection, graceful-shutdown recipe, transcript
  adapter. Adding a harness = one file. An integration without a transcript adapter
  cannot support `agent logs`.
- **Store** — sqlite (WAL); rows for agents, loops, loop runs, events. No files beyond
  the db and loop-run logs: v2 has no run directories.

## 3 · Core concepts

### 3.1 The owned session

All orcr agents live in one dedicated herdr session (default name `orcr`, config
`[herdr] session`). The user's daily herdr session never sees a subagent pane.

- First use auto-starts the session's server headless; `herdr --session orcr` opens the
  native UI over it — this *is* the monitoring dashboard.
- `orcr agent attach` wraps `herdr agent attach`, which streams a pane's terminal into
  the current terminal from anywhere — no session switching.
- `--session <name>` on `agent` commands targets other sessions explicitly: orcr can
  still list / send / read / attach agents the user started by hand in their own
  session (`agent ls --session default`); it just never *places* agents there
  unasked.
- Because orcr owns the session outright, there are no ownership markers, no
  foreign-vs-managed distinctions inside it, and gc/kill have a trivially safe blast
  radius.

Within the session, herdr's hierarchy is used as follows. (herdr facts: workspaces are
per-session, purely visual containers with a label and no cwd; only panes have a cwd
and a process.)

| herdr level | orcr's use |
| --- | --- |
| workspace | = the agent's **level-1 group segment**, displayed via the transform (§3.2): all of `phase_1.*` → workspace "Phase 1"; each loop's runs → one workspace per loop. Ungrouped agents → `default`. GC-parked agents → `idle`. herdr removes empty workspaces automatically. |
| tab | one per agent; label = remaining group path + name/id ("Review / File 1 · a42") |
| pane | the agent's TUI; cwd = caller's cwd or `--cwd`; `pane_id` is the stable internal handle |

### 3.2 Groups

Groups organize agents; they are **strings, not entities** — never created, never
deleted, no registry. A group exists exactly as long as some agent (live or historical)
carries it.

- **Grammar**: dot-separated segments, each `[a-z0-9_]+`. Limits: ≤ 6 segments, ≤ 64
  chars/segment, ≤ 256 chars total. Violations are rejected at spawn with a clear
  error, never truncated.
- **Display transform** (derives human labels from machine slugs; no stored metadata):
  underscores → spaces, words title-cased, dots → " / ".
  `phase_1.review.file_1` → **Phase 1 / Review / File 1**.
- **Inheritance — one rule everywhere**: *effective group = `$ORCR_GROUP` prefix `.`
  explicit `--group`* (env alone if no flag; flag alone if no env; ungrouped if
  neither). Every spawn exports the effective group as `ORCR_GROUP` to the agent, so
  its own spawns nest. Consequences:
  - a workflow script exports `ORCR_GROUP=refactor` once (or SDK `withGroup`) — every
    agent spawned inside, however deep, lands under `refactor.…`;
  - an agent in `refactor.phase_1` spawning with `--group file_1` yields
    `refactor.phase_1.file_1`; with no flag it inherits `refactor.phase_1` verbatim;
  - a loop sets the prefix to `<loop_name>.<unix_ts>` per run (§6);
  - escaping a prefix = unset the env (deliberate, rare; no flag).
- **Prefix operations**, matched on segment boundaries (`phase_1` never matches
  `phase_10`): `agent ls|wait|kill --group <prefix>`.
- **Same name = same group, by definition.** No auto-suffixing. Scripts wanting per-run
  isolation stamp their own suffix (`--group "refactor.$(date +%s)"`).

### 3.3 Lifecycle (GC)

One policy replaces v1's auto-close/keep/reaper trio: `--gc auto|immediate|never`.

| mode | behavior |
| --- | --- |
| `auto` (default) | idle for `idle_after` (5m) → pane moved to the `idle` workspace → `kill_after` (10m) more → graceful kill, memory released. An inbound `send` at any point moves the agent back to its workspace and resets both clocks. |
| `immediate` | killed the moment its work completes (first stable idle). For one-off subagents — nothing lingers. |
| `never` | exempt from parking and reaping; lives until an explicit `agent kill`. For pinned long-livers (heartbeat agents). |

A stuck-*working* agent never goes idle, so `--timeout <dur>` (config default `30m`)
remains the safety valve; on expiry the agent is killed (`exit_reason: timeout`).

### 3.4 Concurrency

- **Global cap** `concurrency.max` (default 25) — RAM protection; heavy TUIs at 100×
  will take a machine down.
- **Per-harness caps** beneath it (e.g. `claude = 10`, `pi = 50`); a spawn needs a free
  slot in both.
- Over-cap spawns get status `queued` and are promoted FIFO as slots free
  (daemon-driven; reconciled by the next CLI invocation when the daemon is down).
  `wait` on a queued agent waits through the queue; `kill` dequeues.
- Loops have a separate, unrelated knob: `--concurrency` caps concurrent *runs of that
  loop* (§6).

### 3.5 Identity & env contract

- Ids: agents `a<N>`, loops `l<N>` — monotonic counters, never reused. Loop runs are
  identified by their unix timestamp within the loop.
- `--name` gives an agent a label unique among *live* agents (dead names are
  reusable); needed for send-by-name (heartbeats). Typed-id shapes (`^[al]\d+$`) are
  rejected as names.
- Env injected into every agent pane and loop command:

  ```
  ORCR_ID        the agent's id (a7)
  ORCR_GROUP     the agent's effective group (prefix for its own spawns)
  ORCR_PARENT    spawning agent's id, if any
  ORCR_SESSION   the owned herdr session name
  ORCR_LOOP      loop id      (loop-run commands and their descendants)
  ORCR_RUN       run unix_ts  (same)
  ```

### 3.6 Completion discipline (internal, hard-won)

herdr reports agent state (`working / idle / blocked / unknown`). Completion is only
ever **`working` observed since the watcher started (small grace window per harness),
then stable `idle`** — a first idle without observed working is never completion. This
guards the send-then-wait race (agent hasn't started yet; a naive wait would return on
the stale idle). Blocked (harness asking a question, limit screen) surfaces as status
`blocked`, exit 4 for waiting callers, and is steerable via `send`.

Other carried-forward herdr driver rules: input delivery is two calls (send-text →
~1s → enter — never one); multiline text is submitted safely per harness; `pane_id` is
the only stable handle after launch; herdr `--timeout` values are milliseconds and
never leak into orcr's user surface.

## 4 · CLI

Three nouns. All verbs support `--json` (exactly one envelope object on stdout —
`{"ok":true,"result":…}` / `{"ok":false,"error":{code,message,details}}` — logs to
stderr). Exit codes: `0` ok · `2` environment · `3` timeout · `4` blocked · `5` killed
· `6` not found · `7` state conflict · `1` other. Durations always carry units
(`45s`, `20m`, `3h`); bare numbers are rejected.

### 4.1 agent

```
orcr agent run    -a <harness> [-p <text> | -f <file|->]
                  [--name <label>] [--group <path>] [--gc auto|immediate|never]
                  [--model <m>] [--effort <e>] [--cwd <dir>] [--timeout <dur>]
                  [--wait] [--session <s>] [--json]
orcr agent send   <id|name> [<text> | -f <file|->] [--json]
orcr agent logs   <id|name> [--last-response] [--tail <n>] [--follow] [--json]
orcr agent wait   <id|name>... | --group <prefix>  [--any] [--timeout <dur>] [--json]
orcr agent attach <id|name> [--takeover]
orcr agent kill   <id|name>... | --group <prefix>  [--json]
orcr agent ls     [--group <prefix>] [--all] [--status <s>] [--harness <h>]
                  [--session <s>] [--json]
```

**run** — spawns the TUI in the owned session, delivers the first message, prints the
new id, returns. `--wait` blocks through the first completion and prints the final
response from the transcript (sugar for `run → wait → logs --last-response`).
Placement per §3.1, admission per §3.4, group per §3.2, gc per §3.3. If the owned
session's server isn't running it is started headless and reported
(`herdr_server: existing|started`). *(Open: async-default is a deliberate reversal of
an earlier wait-by-default decision — flagged for owner confirmation.)*

**send** — types the text into the agent's TUI and submits, whatever state the agent is
in (harness TUIs queue mid-turn input natively; there are no steer/turn modes).
Sending to a GC-parked agent un-parks it. Dead target → exit 6/7 with status detail.

**logs** — reads the harness's **native transcript** via the per-harness adapter
(structured turns, tool calls, token counts where available). `--last-response` prints
just the final assistant message — the standard fan-in read. `--follow` streams. Works
on hand-started agents too: herdr's detection exposes each pane's `agent_session` id,
which locates the transcript for agents orcr didn't spawn. No pane scraping.

**wait** — all-of across the listed targets, or everything under `--group`; `--any`
returns on first completion. Timeout → exit 3 with `completed/pending/blocked`
buckets; blocked target → exit 4 naming it.

**attach** — wraps `herdr agent attach`: streams the pane into the current terminal
from anywhere (inside or outside herdr), detach returns. Observe by default,
`--takeover` claims input.

**kill** — graceful per-harness shutdown recipe (~5s deadline) → pane close; status
`killed`; history remains. `--group` = subtree cleanup in one command; queued agents
are dequeued; `--gc never` agents exit exactly this way.

**ls** — live + queued agents rendered as the group tree (display-transformed
headings). `--all` includes ended agents (the history view — including every past loop
run, queryable by group). `--status blocked` = who needs a human. `--session <s>`
lists herdr-detected agents in other sessions.

### 4.2 loop

```
orcr loop create ("<cron>" | --every <dur> | --once-at <time>)
                 [--name <n>] [--concurrency <n>] [--overlap queue|skip]
                 [--timeout <dur>] [--json] -- <command…>
orcr loop ls | logs <id> [--run <ts>] | pause <id> | resume <id> | rm <id>
```

Durable cron for **any command** — shell one-liner, script, binary. The command spawns
agents via CLI/SDK like any other caller; the loop owns *time only*: no harness flags,
no prompts, no judge logic, no stop-condition DSL.

- **Cadence**: five-field cron (stored UTC; confirmation echoes local + UTC) ·
  `--every <dur>` · `--once-at <time>` (fires once then ends; re-arm via
  `resume --once-at`). Missed fires while the machine slept are skipped and logged.
- **Auto-grouping**: each run executes with `ORCR_GROUP=<loop_name>.<unix_ts>` — the
  script's own `--group phase_1.review` nests to
  `nightly.1752300000.phase_1.review`. Level-1 = loop name → each loop gets its own
  workspace; `agent ls --all --group nightly` is the loop's full agent history.
- **Concurrency**: `--concurrency` caps concurrent runs of this loop (default 1). At
  the cap, `--overlap queue` (default) holds **at most one pending tick** — further
  fires coalesce into it ("run again when done", never N identical queued
  invocations); `skip` drops the tick with a log line.
- **Per-run timeout**: `--timeout` kills an overrunning command; the daemon then kills
  the run's agents (`agent kill --group <name>.<ts>`).
- **Stop conditions live in the script**: it can end its own loop —
  `orcr loop rm $ORCR_LOOP`.
- `loop logs` = each run's captured stdout/stderr (`--run <ts>` selects one run).

### 4.3 daemon

```
orcr daemon start | stop | status
```

Auto-starts on demand (first `loop create`, first queued agent, first GC deadline).
Owns loop firing, GC parking/reaping, queue promotion, timed-out-run cleanup, and
store↔herdr reconciliation (panes gone → agents marked `lost`; unknown panes in the
owned session → closed). `daemon status` is the health probe: version, store path,
herdr binary/server/session reachability, live/queued/blocked counts, loop schedule
and next fires. (`herdr server` is the substrate process; `orcr daemon` is the
supervisor — two words, two processes.)

## 5 · SDK (TypeScript)

A thin typed wrapper over `orcr … --json` — no private protocol; anything the SDK does,
a shell script can do. Published as `@orchestratr/sdk` (name TBD). Python deferred.

```ts
import { orcr } from "@orchestratr/sdk";

// spawn — returns a handle immediately (agent run)
const a = await orcr.agent.run({
  harness: "codex",            // required
  prompt: "…",                 // -p   (or promptFile)
  name?, group?, gc?,          // --name --group --gc
  model?, effort?, cwd?, timeout?, session?,
});

a.id; a.name; a.group;         // resolved identity
await a.wait({ timeout? });    // agent wait <id>
await a.send(text);            // agent send <id>
await a.logs({ tail?, json? });// agent logs <id>
await a.lastResponse();        // agent logs <id> --last-response → string
await a.kill();                // agent kill <id>

// collections
await orcr.agent.wait([a, b], { any?, timeout? });
await orcr.agent.wait({ group: "refactor.phase_1" }, { timeout? });
await orcr.agent.ls({ group?, all?, status?, json: true });
await orcr.agent.kill({ group: "refactor" });

// the one-liner: run + wait + lastResponse, gc immediate
const answer: string = await orcr.ask({ harness: "claude", prompt: "…" });

// grouping — exports ORCR_GROUP for every spawn inside fn (composes with the env prefix)
await orcr.withGroup("refactor", async () => { … });
// withGroup(path, { killOnThrow: true }) → agent kill --group <path> if fn throws

// durable scheduling — the process can exit afterwards
const l = await orcr.loop.create({ every: "30m", name: "burn_down",
                                   command: "npx tsx burn-down.ts" });
await orcr.loop.rm(process.env.ORCR_LOOP!);   // a loop-run script ending its own loop
```

Error mapping: non-zero exits become typed errors carrying `{ code, message, details,
exitCode }` from the JSON envelope. All SDK calls set `--json` internally.

**The file convention** (replaces v1's managed response files): when a step needs a
guaranteed-format answer, the prompt says where to write it — *"write your findings to
./out/security.md, then say DONE"* — and the script reads the file. Agents are good at
writing files; the CLI doesn't manage them. `ask()`/`lastResponse()` cover the casual
cases via transcripts.

Canonical patterns (classify-and-act, fanout-and-synthesize, adversarial verification,
generate-and-filter, tournament, loop-until-done + durable handoff) ship as documented
recipes in the SDK docs; all six are expressible with the surface above.

## 6 · Execution details

### 6.1 Spawn pipeline (`agent run`)

1. Load config; open store; discover herdr; resolve profile.
2. Resolve effective group (env prefix + `--group`); validate grammar/limits.
3. Admission: free slot in global AND harness caps → proceed; else insert `queued`
   (FIFO) and return the id (a queued spawn still prints its id immediately).
4. Allocate `a<N>`; ensure owned session server; ensure workspace for the level-1
   segment (create with display-transformed label if absent).
5. `herdr agent start` in a new tab: harness argv from the profile (bypass flag,
   model/effort mapping), env contract injected, `--no-focus`.
6. Run the profile's startup recipe (dismiss update modals etc.).
7. Deliver the first message (two-call input rule).
8. Record rows + events. If `--wait`: watch to completion (§3.6), print
   `logs --last-response`, exit by outcome.

### 6.2 GC engine (daemon)

Tick every ~30s: for each live `gc: auto` agent, if idle ≥ `idle_after` → move pane to
the `idle` workspace (herdr `pane move`); if parked ≥ `kill_after` → graceful kill.
For `gc: immediate` agents the completion watcher kills at first stable idle. `send`
to a parked agent moves it back and resets clocks (performed by the CLI inline, not
the daemon). All moves/kills are recorded as events.

### 6.3 Loop scheduler (daemon)

Computes next-fire per loop (cron in UTC / interval from last completion /
`--once-at`). On fire: if running-count < `--concurrency` → start a run; else apply
overlap policy (`queue`: set the single pending-tick bit; `skip`: log). A run =
executing the command with the env contract (`ORCR_LOOP`, `ORCR_RUN`,
`ORCR_GROUP=<name>.<ts>`), stdout/stderr captured to the run log. On exit: record
status; if a pending tick is set and slots are free, fire immediately. On per-run
timeout: kill the command's process group, then `agent kill --group <name>.<ts>`.

### 6.4 Reconciliation

On daemon start and periodically: diff store ↔ herdr. Agents whose panes vanished →
status `lost` (event recorded). Panes in the owned session unknown to the store →
closed. Queued agents whose slots freed while no daemon ran → promoted by the next CLI
invocation (reap-on-touch).

### 6.5 Transcript adapters

Per-harness modules that locate and parse the harness's native session files (e.g.
claude/codex JSONL) into a common shape: ordered messages with roles, tool calls,
token counts where present. Used by `agent logs`, `--last-response`, `--wait`'s
response printing, and the SDK's `ask()`. Location strategy: the pane's
`agent_session` identity as reported by herdr detection (works for hand-started
agents), falling back to harness-specific discovery from the agent's cwd. pi and
opencode adapters are required work items — an integration without an adapter cannot
support `logs`.

## 7 · Store

sqlite, WAL, `user_version` guard. No run directories in v2.

```
agents:    id PK (a<N>), name, group_path, parent_id, harness, model, effort,
           gc_mode, cwd, session, pane_id, tab_id, workspace_id,
           agent_session_kind, agent_session_value,        -- transcript pointer
           status (queued|starting|working|idle|parked|blocked|done|killed|lost|timeout),
           exit_reason, timeout_s, created_at, ended_at
loops:     id PK (l<N>), name, cadence_kind (cron|every|once), cadence_value, tz,
           command_json (argv), concurrency, overlap, timeout_s,
           status (active|paused|ended), pending_tick INT, created_at, ended_reason
loop_runs: loop_id, run_ts PK, status (running|ok|failed|timeout), exit_code,
           log_path, started_at, ended_at
events:    seq PK AUTOINCREMENT, ts, kind, ref_id, payload_json   -- append-only audit
counters:  name, value                                             -- id allocation
```

Group queries are prefix scans on `group_path` (indexed; matching on `.` boundaries).

## 8 · Configuration

```toml
# ~/.orcr/config.toml — every key optional; defaults shown
[defaults]
harness = "claude"
model   = ""            # empty = harness default
effort  = ""
timeout = "30m"         # stuck-working safety valve

[herdr]
bin     = ""            # empty = $ORCR_HERDR_BIN → $PATH
session = "orcr"        # the owned session; the user's own sessions are never touched

[concurrency]
max    = 25             # global ceiling (RAM protection)
claude = 10             # per-harness caps beneath it (any harness name is a key)
pi     = 50

[lifecycle]
idle_after = "5m"       # idle this long → parked in the idle workspace
kill_after = "10m"      # parked this long → reaped
```

Env: `ORCR_STORE` relocates the store (tests/sandboxes depend on it); `ORCR_SESSION`
overrides the session per invocation; `ORCR_HERDR_BIN` overrides discovery.

## 9 · Open items

1. **`agent run` default: async vs wait** — this spec says async (id printed;
   `--wait` opt-in), reversing an earlier wait-by-default decision, because GC-managed
   lifetimes and group fan-ins make compose-with-`wait` the norm. Awaiting owner
   confirmation.
2. pi / opencode transcript adapters (required for `logs` on those harnesses).
3. Blocked-state detail: how much reason detail herdr exposes per harness (limit
   screen vs question) — best-effort surface in `ls`/`wait`.
4. Later (explicitly deferred): remote hosts, permission policies, Python SDK,
   coordination primitives, worktrees, Windows.
